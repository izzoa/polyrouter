Add the notification-channel delivery core (spec §10.1/§12; invariants 5/6/8/11) — the async, SSRF-guarded, secret-safe pipeline that later producers (#15b `provider_down`/spikes/summaries, #16 budget alerts) emit into. Owner-scoped `notification_channel` CRUD with a safe API view (`hasConfig`, never the decrypted blob); the whole kind config (`smtp` `{host,port,secure,user?,pass?,from,to[]}` or `apprise` `{urls[]}`) is AES-GCM-encrypted at rest under `NOTIFY_CREDENTIALS_SECRET` (gated exactly like `PROVIDER_CREDENTIAL_KEY`). Every server-reached destination is SSRF-validated under a **mode-gated policy that mirrors the local-provider exception**: metadata/link-local is blocked in every mode and never allowlistable; self-host allows the operator's own loopback/private infra (the docker Apprise sidecar and a local SMTP relay work out of the box); cloud blocks loopback/private unless a **port-bounded** `NOTIFY_ALLOWED_ENDPOINTS` entry permits a soft range. The SMTP adapter validates **at connect time** and pins the transport to the validated IP (SNI preserved) so a rebind can't redirect the socket; Apprise targets are scheme-aware (fixed-service schemes allowed, host-bearing schemes and `?smtp=`/`?host=` overrides validated, unknown rejected); `APPRISE_API_URL`'s host is validated in an async `NOTIFY_RUNTIME` factory so a bad value **fails boot** before a worker exists; cloud Apprise is additionally gated on `NOTIFY_APPRISE_EGRESS_CONFIRMED`, enforced **at delivery**. Delivery is a BullMQ pipeline on dedicated fail-fast producer + worker connections: `emit` is O(1), TTL-deduplicated per `(type, scope)` window, and never throws or blocks the caller (bounded even when Redis is blackholed); a fan-out job loads the owner's subscribed channels and enqueues per-channel delivery jobs that carry only `{channelId, ownerId, event}` — the **structured, secret-free** `NotificationEvent` whose title/body the worker renders from non-secret display `fields`. A failing channel is retried with bounded backoff then left failed and logged, isolated from other channels/events. All delivery errors throw only **fixed sanitized codes** (no host/recipient/URL/response), so no channel credential or raw error reaches the job store, logs, `last_test_status`, or the API. `POST /api/notification-channels/:id/test` delivers a `test` event directly for inline feedback. New deps: `bullmq`, `nodemailer`. Producers, the scheduler, the breaker-transition read, and password-reset wiring are #15b.
