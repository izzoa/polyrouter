Make the proxy as observable as what it sells (#21, spec §3.2.6/§14.10): OpenTelemetry traces on the proxied request path and a Prometheus `/metrics` endpoint, both incapable of affecting a request's outcome.

**Traces (off by default — `OTEL_ENABLED`, standard OTLP/HTTP export, batched/async).** Every authenticated inference request produces one trace: a `proxy.request` root (started by a `/v1` Express middleware in `configureApp`, ended on response close — streaming and client aborts included) with children `auth` (agent-key verify, outcome only — never key material), `routing` (route resolution + structural/cascade evaluation, decision-layer/tier/model attributes), one `upstream` span per attempted chain member (provider/model attributes; error status on failure; a client abort marked `canceled`, never a provider error), and `recording.enqueue`. The durable insert is traced at the writer as a `recording.write` batch span **linked** to the originating requests' span contexts (started under `ROOT_CONTEXT` so a threshold-triggered flush never parents to an arbitrary request), with insert give-ups marked. Manual spans only — no auto-instrumentation packages; with tracing off, every call is a no-op. The SDK registers in `main.ts` before Nest and flushes on graceful shutdown. `OTEL_EXPORTER_OTLP_ENDPOINT` is a registered optional URL (malformed fails boot fast; unreachable is never fatal); `OTEL_SERVICE_NAME` defaults to `polyrouter`.

**Metrics (`GET /metrics`, prom-client, default on — `METRICS_ENABLED=false` hides it with a 404).** Instance-level, bounded-cardinality series (prefix `polyrouter_`, labels only ever provider display name / external model id / small enums — never tenant, agent, request ids, or content): request count + duration histogram per protocol/decision-layer/status (emitted at enqueue time, exactly once per finalized inference request, so traffic stays visible through a DB outage); token counters per provider/model/direction (including superseded cascade cheap attempts); **cost counters incremented from the authoritative snapshot cost exactly once per row, only after its batch insert succeeds** (writer retries never re-emit; unpriced/dropped rows emit nothing — invariant 4 intact); upstream attempt count + duration per provider/model with success|error|canceled outcomes (streamed error events and truncation count as error; client aborts as canceled) plus a setup-failure counter for members that failed before any upstream call; **circuit-breaker state gauge** (last observed at admission: closed/half_open/open) and open-transition counts per provider; log-writer dropped-row counts; Node process metrics. Every emit path is exception-safe; the registry is per-app instance.

**Data-plane (additive, no behavior change):** the chain-runner options and `withBreaker`/`withBreakerStream` gain an optional `onBreakerState` observation callback (fired with the state observed at each admission decision) alongside the existing `onOpen`; existing callers are untouched.

**Fix surfaced by the cost metrics:** a successful cascade cheap call is now ledgered (`request_attempt` row + token/cost accounting) even when **every** escalation member fails — previously that billed call vanished from spend on total escalation failure (§7.7 completeness).

Known follow-up (pre-existing, unchanged here): a client disconnect during a stream's pending body read is normalized to `ProviderError('unavailable')` inside the HTTP adapters, so it still counts one breaker failure; the new metrics classify it correctly as `canceled` via the client signal, but the breaker/`provider_down` classification predates this change and is left for its own fix.
