# TODOS вҖ” polyrouter build plan (OpenSpec change breakdown)

**App name: `polyrouter`** вҖ” an open-source, self-hostable LLM router/gateway. Operating rules: [`CLAUDE.md`](./CLAUDE.md); capability contracts: `openspec/specs/`.

This file decomposes the spec into **individually proposable OpenSpec changes**. Each entry is one capability вүҲ one change, listed in dependency order (per spec В§14 / CLAUDE.md build order). Tables owned by a capability land **with that capability's change** (own migration), not up front.

## Per-change workflow (repeat for every entry below)

1. **Propose** вҖ” `/opsx:propose <change-name>` (generates `proposal.md`, `design.md`, `tasks.md`, spec deltas under `openspec/changes/<change-name>/`).
2. **Verify proposal** вҖ” review `proposal.md` + `tasks.md` + deltas against the matching `openspec/specs/` contracts and the CLAUDE.md invariants; DoD scenarios must map to the acceptance criteria listed in the entry.
3. **Clink (codex review)** вҖ” run a codex review via the unison `clink` tool over the proposal artifacts; fold in findings before implementation.
4. **Apply** вҖ” `/opsx:apply` and work `tasks.md` in order. Stop and flag (never reinterpret) any task that contradicts the spec or an invariant.
5. **Verify implementation** вҖ” DoD below: tests green (`npm test`, e2e suites where relevant), `npm run build` passes, lint clean, migration generated if schema changed, changeset added if user-facing. Optionally clink the implementation diff with codex before archiving.
6. **Archive** вҖ” `/opsx:archive` (merges deltas into `openspec/specs/`).
