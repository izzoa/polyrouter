# Design: add-auth-and-identity

## Context

#2 shipped the schema (Better Auth-compatible `user` table with a nullable `role`), the scoped `PersistencePort` (+ `withAdvisoryLock` explicitly reserved for this change), `ensureDefaultTier`, encryption, and Redis. This change adds the two credential planes on top. Constraints: invariant 7 (HMAC agent keys, never bcrypt; slow-hashed passwords), §11.1 (the session guard must resolve a `Principal` for the scoped repositories), §11.3 (throttle auth endpoints; first-admin race guard), the raw-handle privacy requirement from #2, and the `MODE=selfhosted` gating rules.

## Goals / Non-Goals

**Goals:** working email/password + OAuth sessions; `/api` protected by default; exactly one admin under concurrent first signups; default tier at signup; localhost auto-login + seed admin (self-host); Redis-backed auth throttling; fast-verified agent keys with shown-once minting; tenant-scoped Agents CRUD with shared snippets.

**Non-Goals:** proxy routes (#10), dashboard UI (#18), org membership (deferred), email delivery (#15), Better Auth admin plugin.

## Decisions

1. **Better Auth 1.6.23, mounted as Express middleware, with explicit per-plane guarding (codex #2).** `toNodeHandler(auth)` handles `/api/auth/*` before the Nest router. Guarding is **plane-scoped, not a blanket global guard**: `SessionGuard` is bound to `/api/**` only (validating cookies via `auth.api.getSession({ headers: fromNodeHeaders(req.headers) })`); `/v1/**` is the agent-key plane (bound to `AgentApiKeyGuard`, landing with #10 — this change ships the guard and proves it on a test-only probe route mounted under a `/v1`-style path); `GET /api/health` is `@Public()`. Neither credential authenticates on the other plane (a session cookie does nothing on `/v1`; a Bearer agent key does nothing on `/api`) — asserted by e2e. This keeps the two credential planes separate (invariant 7). Alternatives (hand-rolled sessions, passport) rejected: Better Auth is pinned and owns password hashing, OAuth, CSRF.

2. **The Better Auth drizzle adapter is constructed INSIDE the database module (codex #6).** Better Auth's adapter needs a drizzle handle, and #2's raw-handle-privacy requirement says unscoped SQL is unwritable outside the persistence module. So the database module builds the adapter (over its own private pool, restricted to the four identity tables via explicit schema mapping) and exports it as an **opaque `AUTH_ADAPTER` token** — no new raw handle escapes; the auth module receives a ready adapter it cannot use for arbitrary tenant-table SQL. Identity operations are a **separate `IdentityPort` token** (not folded into the tenant `PersistencePort`) exposing **semantic** methods — `ensureFirstAdmin()` (advisory-locked promote-earliest-if-none, returns the admin id), `findAdminUserId()` (for auto-login), `provisionDefaultTier(userId)`, and **`provisionMissingDefaultTiers()`** (advisory-locked; finds and provisions every user lacking a `default` tier — the boot reconciliation needs this because a *later* user's crashed hook leaves a missing tier that a first-admin check would never discover, codex round-2 #2) — rather than a generic `setRole`/`earliestUserId` that could be misused. Raw Pool/drizzle stay private exactly as #2 guaranteed.

3. **Auth-plane tables match Better Auth 1.6.23's shapes completely (codex #7).** `session` (id, token unique, expires_at, created_at, updated_at, ip_address, user_agent, user_id FK cascade); `account` (id, account_id, provider_id, user_id FK cascade, access_token, refresh_token, id_token, access_token_expires_at, refresh_token_expires_at, scope, **password** — the scrypt credential, created_at, updated_at); `verification` (id, identifier, value, expires_at, created_at, updated_at). snake_case columns, indexes on session.user_id / account.user_id / verification.identifier. The adapter receives an **explicit model→table map** (`{ user: users, session: sessions, account: accounts, verification: verifications }`) because our exported consts are plural while Better Auth's model keys are singular. One migration; the generated SQL is diffed against Better Auth's 1.6.23 CLI schema snapshot in review.

4. **Password hashing: Better Auth's scrypt satisfies the "argon2/bcrypt" intent — flagged, not silent.** Spec §3.2.3 names argon2/bcrypt for session passwords; Better Auth uses **scrypt** (also a memory-hard slow KDF). Swapping in argon2 would add a native dependency for no security gain. The spec delta says "slow, memory-hard KDF (scrypt)" and the e2e asserts the stored credential is a salted scrypt string, not plaintext or a fast digest.

5. **`role` is our column, exposed via `additionalFields` with `input: false`** — server-owned, so signup payloads cannot mass-assign admin. The Better Auth admin plugin (ban/impersonation/admin endpoints + extra columns) is out of scope; first-admin is our logic.

6. **First-admin + default-tier: deterministic under advisory lock, AND reconciled at boot (codex #1).** Better Auth 1.6.23 commits the user-insert transaction *before* running queued `after` hooks (it awaits them, but they are post-commit), so a hook crash could otherwise strand a committed user with no admin/tier. Two layers close this: (a) the `user.create.after` hook runs `IdentityPort.ensureFirstAdmin()` (`withAdvisoryLock(FIRST_ADMIN_LOCK)`: promote the earliest user iff no admin exists — idempotent, so any interleaving of N concurrent signups converges to exactly one admin) and `provisionDefaultTier(user.id)` (idempotent per #2's `ensureDefaultTier`); (b) a **boot-time reconciliation**, also advisory-locked, calls `ensureFirstAdmin()` **and `provisionMissingDefaultTiers()`** to heal any zero-admin *or* any user's missing-default-tier state (including a non-first user whose hook crashed) before the guard serves traffic. During any brief post-commit-pre-promotion window the system is **fail-closed** — protected routes 401 and auto-login refuses (no admin resolvable) — never fail-open. Fault/restart recovery is tested.

7. **Localhost auto-login = virtual admin principal, anchored on non-reachability (codex #3, round 2).** Topology checks alone can't tell a direct loopback client from a same-host reverse proxy (its upstream socket is genuinely loopback, it rewrites `Host` to localhost, and non-browser clients omit `Origin`/`Sec-Fetch-Site`). So the **load-bearing gate is that the server is not network-reachable**: auto-login applies ONLY when ALL hold — `MODE=selfhosted`; **`BIND_ADDRESS` resolves to a loopback address** (if the operator bound to `0.0.0.0`/a routable address to put a proxy or the network in front, auto-login is OFF and a real login is required); **no forwarding headers present** (`X-Forwarded-*`/`Forwarded` ⇒ something is proxying ⇒ refuse — a direct loopback client never sends them); the raw socket peer is loopback; the `Host` header is loopback (DNS-rebinding defense); the request is same-origin (`Origin`/`Sec-Fetch-Site` absent or the exact dashboard origin); `trust proxy` stays OFF; and an admin exists (else 401). CORS is pinned to the exact Vite dev origin. Threat boundary: a genuinely local, loopback-bound instance trusts same-host processes (spec §11's premise); the moment it is network-bound or proxied, auto-login disables and credentials are required. (A file-based bootstrap-token alternative was considered but rejected as friction the loopback-bound gate makes unnecessary.) Cloud: never. E2e proves auto-login on a loopback-bound selfhosted app, and its refusal when a forwarding header, foreign Host, or foreign Origin is present.

8. **`SEED_DATA` and fixed dev secrets require a loopback-bound instance, never network-reachable (codex #4, round 2).** Both the seed admin and the fixed `polyrouter-dev-*` fallback secrets are gated on `MODE=selfhosted` **and** `NODE_ENV!==production` **and `BIND_ADDRESS` loopback** — a well-known credential or a fixed HMAC/session secret must never exist on any instance the network can reach. A boot that is network-bound (or production) while `SEED_DATA=true` or while relying on fallback secrets **fails fast, naming what to set**. The seed creates `admin@localhost` via the normal signup API; its **password is never logged** (invariant 8) — the boot logs only the seeded email and points at the dev README for the documented password.

9. **Auth throttling: atomic Redis fixed-window, unambiguous per-mode failure policy (codex #5, round 2).** A **Lua script** does INCR + set-expiry-only-on-first-hit + return-TTL atomically (a bare `INCR`+`EXPIRE` can leak a non-expiring key on a mid-pair crash). Limits, per client IP per **actual Better Auth 1.6 route**: `/sign-in/*` 10/min, `/sign-up/*` 5/min, and both `/request-password-reset` and `/reset-password` 3/5min → 429 with `Retry-After`. Client IP: the raw socket peer, **and `X-Forwarded-For`'s last hop is consulted only when that immediate peer is itself within a configured `TRUSTED_PROXY_CIDRS` entry** (otherwise the header is ignored — unspoofable). **Redis-outage behavior is one exact rule, not a hedge**: the limiter falls back to a **per-instance in-process fixed-window limiter with the identical per-route limits**. Under `selfhosted` (single instance) that is equivalent to normal enforcement; under `cloud` it degrades from global to per-instance counting (a distributed attacker gets at most N× the limit across N instances during the outage — bounded and logged, never fully open and never a total lockout). The mode difference is only the log severity and that cloud additionally alerts. Counts stay correct across instances while Redis is up (§3.2).

10. **Agent keys: `poly_` prefix + 32 base64url payload chars (24 random bytes); stored prefix = `poly_` + the first 12 PAYLOAD chars (codex #10).** That's 72 bits in the prefix (not 7 chars/42 bits — the literal-first-12 reading was ambiguous), collision-free in practice and unique-indexed since #2 (mint retries on the vanishingly-rare collision); the full token carries 192 bits. Stored: `api_key_hash = HMAC-SHA256(key, API_KEY_HMAC_SECRET)` hex + prefix — never the key. Verification: prefix lookup → `timingSafeEqual` on HMAC bytes — two fast hashes, no KDF (invariant 7). **Proof that no KDF is on the path is a deterministic unit assertion** (the verify path calls only HMAC + `timingSafeEqual`, asserted by construction/spy), with a non-gating benchmark as corroboration. Create/rotate responses carry `Cache-Control: no-store`. The `IdentityPort` gains `agentAuth.findByPrefix`/`touchLastUsed` — authentication, deliberately unscoped, returning only id/owner/hash/prefix.

11b. **`last_used_at` is coalesced, not one-write-per-request (codex #8).** A per-agent in-memory throttle stamps at most once per N seconds (acceptable lag documented; the value is advisory UI data, not billing), the update is `.catch()`-guarded so a rejected promise can never crash the process, and it never blocks the response. Tests poll for the eventual update.

11. **Agents CRUD rides the scoped repository**: list/create/rotate/delete under `/api/agents` with the session principal; responses never include `api_key_hash`; the full key appears exactly once in create/rotate responses alongside a connection snippet. **Snippets and the harness list move to `shared/src/harness.ts`** (root entry — browser-safe) as the §2.1 canonical data; the dashboard prototype's local copy is replaced by the shared one (visible drift eliminated).

12. **Secrets policy: required in production, fixed dev fallbacks otherwise.** `BETTER_AUTH_SECRET` and `API_KEY_HMAC_SECRET` validate as 64-hex when present; when absent — `NODE_ENV=production` fails the boot naming the variable; development/test fall back to fixed, obviously-labeled dev constants with one loud warning (ephemeral randoms would invalidate agent keys and sessions on every dev restart). Values are never echoed (config framework rule).

12. **Password reset: token flow now, email delivery explicitly deferred to #15 (codex #9).** Better Auth's `sendResetPassword` callback delivers the reset URL; #3 wires the callback to a logging stub that records that a reset was requested **without ever logging the token**, and the spec/TODOS state delivery lands with #15 (SMTP/Apprise). A TODOS.md note is added to #15 so the dependency is tracked, not forgotten. Rate limiting on the reset routes ships here regardless.

## Risks / Trade-offs

- [Rate limiter failure policy] → mode-aware (decision 9): self-host fails open (availability), cloud fails closed with an in-process backstop (brute-force protection is mandatory). Atomic via Lua; proxy-aware IP only with explicit `TRUSTED_PROXY_CIDRS`.
- [Auto-login trusts the local host] → its §11 purpose, hardened (decision 7): loopback socket + loopback Host + same-origin + admin-exists, all under `MODE=selfhosted`, CORS pinned to the dev origin. Same-host processes are the documented trust boundary; proxies and browsers are not trusted.
- [Post-commit first-admin window] → fail-closed during it (decision 6) and healed by boot reconciliation; never a fail-open gap.
- [Two pg pools (tenant + auth)] → the auth adapter is built inside the database module over identity tables only and exported opaquely (decision 2); no new raw handle escapes, so #2's isolation guarantee is intact.
- [Better Auth internals move between minors] → pinned `1.6.x`; integration points verified against 1.6.23 docs/source (after-hooks are post-commit — decision 6); e2e exercises real endpoints; generated schema diffed against the 1.6.23 CLI snapshot.
- [Dev seed/secrets] → seed and fixed dev constants exist only when `selfhosted` AND non-production; production refuses to boot without real secrets or with `SEED_DATA`; the seed password is never logged.
