# Design: add-database-and-tenancy

## Context

The monorepo scaffold (#1) and dashboard prototype (P) exist; there is no persistence. This change is the data foundation for every backend change (#3–#22). Constraints: pinned stack (Drizzle NOT TypeORM, PostgreSQL 16, Redis), CLAUDE.md invariants 5 (central tenant scoping — "no by-id fetch without an ownership guard") and 8 (credentials encrypted at rest, never logged), the workspace dependency matrix from #1 (`data-plane → shared` only), and #1's config framework (each capability registers its own vars).

## Goals / Non-Goals

**Goals:** §5 identity/config schema with migrations-on-boot; a tenant-scoping guard that later changes *cannot bypass by accident*; a reusable IDOR test harness; encrypt-at-rest primitives; a Redis client module; dev/test database infrastructure.

**Non-Goals:** auth (#3), endpoints, org/team feature, feature-owned tables (ModelPrice/RequestLog/NotificationChannel/Limit), Redis counters/queues/breakers, seeding beyond schema.

## Decisions

1. **Schema lives in `@polyrouter/shared/server`, not `control-plane/src/entities` (deviation from §4, flagged).** The data-plane must read Providers/Models/Tiers (#10) and write RequestLog (#11), and the dependency matrix forbids `data-plane → control-plane`. Drizzle table definitions are types + metadata with no runtime coupling, so `shared` is the correct home; the §3.3 extraction then needs no schema move. The `./server` entrypoint (second tsup entry, own `exports` condition) keeps node-only code (drizzle-orm, node:crypto) out of the frontend bundle. The lint guard covers **every import form** — package subpath, deep source path, relative path, and built output — and the root entrypoint never re-exports server symbols (asserted by test). Migrations output stays at `control-plane/src/database/migrations` per §4 (the control plane owns running them in-baseline). **The data-plane persistence seam is defined now**: `shared/server` exports a persistence injection token **typed as the scoped `PersistencePort`** (repositories, join-scoped accessors, `ensureDefaultTier`, a narrow `users.count()` infra accessor — no `query`/`execute`/Pool/drizzle member anywhere on the type); the control-plane database module *provides* against that token, so #10/#11's data-plane module injects persistence by token without ever importing control-plane — proven by an integration test that resolves the token from a consumer module knowing only `shared`.

2. **Driver: `pg` (node-postgres) Pool + `drizzle-orm/node-postgres`.** #3 needs transactions and an advisory lock (first-admin race, §11.3); `pg` is the battle-tested path for both. Migrations run programmatically on boot via drizzle's migrator against the same pool, before the HTTP server binds — a failed migration is a failed boot (fail-fast, consistent with app-config). `drizzle-kit generate` produces SQL under `src/database/migrations/`; the **build copies the migrations folder into `dist`** and the migrator resolves it relative to the built entrypoint, so the production build is self-contained (verified by an e2e against the built app). `db:generate`/`db:migrate` exist as package scripts **and root forwarding scripts**, matching CLAUDE.md's command set verbatim.

3. **Ownership model: `owner_user_id` (FK → user, required) + `org_id` (nullable, unused stub) on every owned table — a deliberately user-only foundation.** `Principal` is the union `{ kind: 'user', userId } | { kind: 'org', orgId }` (shared/server type) so downstream signatures never change, but **this change implements user scoping only**: the predicate helper throws on org principals with a pointer to the deferred org change, which owns the real work (ownership-union schema migration, constraint changes, predicate branch, membership). No "single-site change" claim — org support is honestly a schema-and-guard change, centralized where it will happen.

4. **The guard is a scoped-repository factory, not a convention — and raw handles are not exported.** `createOwnedRepository(db, table)` works over tables satisfying an `OwnedTable` shape (id + owner_user_id + org_id columns) and exposes ONLY principal-taking methods: `findById(principal, id)`, `list(principal)`, `insert(principal, values)`, `update(principal, id, patch)`, `remove(principal, id)` — every one appends the ownership predicate; update/remove return affected counts so "someone else's id" and "nonexistent id" are the same not-found. **`id` and ownership columns are immutable through the API**: insert sets the owner from the principal (never caller input) and update strips/rejects `id`/`owner_user_id`/`org_id` at both the type level and runtime — no cross-tenant transfer by patch. **The database module keeps the raw Pool/drizzle providers private**; its only exports are the `PersistencePort` token plus two privileged facilities — `withTransaction(fn)` and `withAdvisoryLock(key, fn)` (for #3's first-admin race) — **whose callbacks receive a transaction-bound `PersistencePort`, never a raw handle**, so even privileged code cannot issue unscoped SQL. Tests assert the raw providers do not resolve from outside the module and that no exported surface exposes `query`/`execute`/Pool/drizzle members. Instances for Agent/Provider/Tier/RoutingRule ship now as the pattern. **Parent-owned rows are tenant-checked at mutation time, not just read time**: Model (via provider) and RoutingEntry (via tier + model) get accessors whose inserts atomically validate parent ownership (ownership-filtered insert inside a transaction, backed by FKs so a racing parent delete fails closed), whose updates treat ownership-defining FKs (`model.provider_id`, `routing_entry.tier_id`/`model_id`) as immutable, and whose deletes are join-scoped single statements — user A can neither create a model under B's provider, link a tier entry to B's model, nor repoint a row into another tenant.

5. **User table is Better Auth-shaped** (`id` text PK, `name`, `email` unique, `email_verified`, `image`, `created_at`, `updated_at`) so #3 points the Better Auth Drizzle adapter at it without a rename migration; auth-plane tables (session/account/verification) land in #3. Text ids via `crypto.randomUUID()` app-side defaults, matching Better Auth's id style.

6. **Encryption: AES-256-GCM, pure functions, versioned envelope.** `encryptSecret(plaintext, key)` → `poly-enc:v1:<iv>:<tag>:<ciphertext>` (base64 fields); `decryptSecret(envelope, key)` verifies the auth tag (tamper ⇒ throw). Keys are 32-byte-hex strings **passed in by callers** — no env var lands here; #7/#15 register their own key vars (spec §12 names `NOTIFY_CREDENTIALS_SECRET`; #7 will introduce its equivalent). Error messages name the operation only — never plaintext, key, or ciphertext (invariant 8), unit-asserted.

7. **Redis: `ioredis`**, because BullMQ (#15, pinned in CLAUDE.md) requires it. A small Nest module provides the client (`lazyConnect`, retry strategy) and closes it on shutdown. Nothing else — counters/breakers/queues belong to their owning changes.

8. **Config defaults match dev infrastructure.** `DATABASE_URL` (default `postgresql://polyrouter:polyrouter@localhost:5432/polyrouter`) and `REDIS_URL` (default `redis://localhost:6379`, the §12 example) — URL-validated, protocol-checked, registered under `database`/`redis` namespaces. Secrets get no defaults ever; these are non-secret local endpoints (credentials only meaningful for the dev container), so `npm ci && docker compose -f docker-compose.dev.yml up -d && npm run dev` works with zero env files. `docker-compose.dev.yml` (postgres:16-alpine + redis:7-alpine, polyrouter dev credentials) is **developer/test infrastructure** — #22's product compose is a separate artifact.

9. **The IDOR e2e harness runs against real Postgres** (the dev compose), not a mock — predicate bugs are SQL bugs. `test/tenancy/harness.ts` provides `createTestPrincipal()` (inserts a user row directly; no auth exists yet) and boots the Nest app with migrations. If the database is unreachable, the suite **fails with instructions** (`docker compose -f docker-compose.dev.yml up -d`) rather than skipping — a silently-skipped isolation suite would fake the DoD.

10. **Downstream-critical constraints live in the schema, not just app code.** `routing_entry.position` is **`INTEGER NOT NULL`** (PostgreSQL CHECKs pass NULL and uniques admit multiple NULLs — nullability would void the cap) with `CHECK (position BETWEEN 0 AND 4)` plus `UNIQUE (tier_id, position)`, and non-null `tier_id`/`model_id` — the §7.4 **five-models-per-tier cap, total** (canonical reading; §5's "cap 5 fallbacks" wording and TODOS.md #9 are reconciled to five total), concurrency-safe at the database so #9's CRUD can't race past it. `model` gets `UNIQUE (provider_id, external_model_id)` so #7's catalog sync is idempotent, and its columns are enumerated per §5 (context_window, tools/vision/reasoning flags, per-1M prices, is_free, last_synced_at). `tier` gets `UNIQUE (owner_user_id, key)`, which makes `ensureDefaultTier(principal)` an idempotent, concurrency-safe upsert — **the provisioning contract ships and is tested here; #3 invokes it at user creation** (spec §5 "ships with `default` seeded").

## Risks / Trade-offs

- [Schema-in-shared drifts from §4's tree] → deviation documented in proposal + this file; §4's *intent* (extractable data-plane) is the reason for it.
- [Localhost DB defaults could mask a prod misconfiguration] → defaults are loopback-only; #22's container sets real values explicitly, and connect failures fail boot loudly with the attempted host.
- [Guard bypass via raw `db`] → raw Pool/drizzle providers are private to the database module (non-export asserted by test), and the privileged facilities hand out a transaction-bound scoped port rather than a raw handle — there is no exported surface on which unscoped SQL can be written. Residual risk is someone adding a new export; the surface test pins the export list.
- [drizzle-kit majors move fast] → generated SQL is committed; CI never regenerates implicitly.
- [Model/RoutingEntry indirect ownership adds join cost] → negligible at config-table scale; correctness first (§11.1).

## Migration Plan

First schema migration in a repo with no users — nothing to migrate. Rollback = drop the dev database (`docker compose -f docker-compose.dev.yml down -v`).

## Open Questions

None blocking. (Whether `db:generate` should be CI-verified as drift-free is a #21/CI-era concern.)
