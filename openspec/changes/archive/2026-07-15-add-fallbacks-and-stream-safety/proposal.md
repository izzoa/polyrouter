# Add fallback chains + mid-stream safety (completes the shippable core)

## Why

#10 routes to a tier's **primary** model only; a provider hiccup fails the request. The spec's reliability core (spec §7.4, §6.3, §8, §3.2.5; invariants 3, 12) is the **fallback chain**: walk the tier's ≤5 ordered models until one succeeds, prefer flat-rate **subscription** quota before paid API keys (§8), skip a rate-limited/down provider **fast** via a Redis circuit breaker (§8, §3.2), and record which model served and why the others failed. Streaming makes this subtle: a transparent fallback is only safe **before the first token** — #10 built the commit boundary for exactly this, and #12 puts the chain behind it. This is the last change before the ⛔ review gate; after it, #10–12 are the shippable core.

The bar (spec §15): fallbacks fire on primary failure and the request still succeeds; a failure *before* the first token falls back transparently; a failure *after* streaming has begun terminates the stream with a clear error and is **never** silently swapped mid-response.

## What Changes

- **Ordered chain resolution** (`data-plane/src/routing/resolve.ts`, additive): `RouteDecision` gains `chain: RouteTarget[]` — a tier resolves to **all** its entries in position order (chain[0] = the primary), an explicitly-named model or a `model:` rule target to a single-element chain (no fallback). No client-visible change to #10's primary selection.
- **Fallback walk in `ProxyCore`** (`data-plane/src/proxy/core.ts`): `runBufferedChain` and `openStreamChain` walk the chain in **configured order** — each attempt's adapter built **lazily** (so a broken later fallback can't fail a healthy primary) and wrapped in #6's **circuit breaker** — trying the next member on a **fallback-eligible** failure (classified on the RAW error: a retryable `ProviderError`, a `ProviderCircuitOpenError` skip, or a member build failure; a `bad_request` or a client cancellation is **not** retried). Non-streaming retries until one returns; streaming retries **until the first successful event commits**, then streams committed — a post-commit upstream error is the #10 terminal error frame, **never** a mid-response model swap. The single-attempt commit gate is refactored to a signal-aware factory (`openAttemptStream`) so its first-event timeout can cancel the upstream; the buffered path gains a disconnect `AbortController`.
- **Configured-order chain, subscription via configuration** (`ProxyService`, §8): the chain is NOT auto-reordered (§7.4/§5 define an explicit primary/fallback order); §8's subscription preference is achieved by configuring the subscription model earlier, so its quota is used first and a limit falls through to paid.
- **Circuit breaker wiring + a stream-event fix**: a shared `CircuitBreaker` over #6's `RedisBreakerStore` (via `RedisModule`, bounded by a fast-fail deadline) with an `InMemoryBreakerStore` fallback (degrades gracefully if Redis is down/slow — invariant 1). #6's `withBreakerStream` is fixed to **settle the classified outcome before yielding an error event** so the commit gate cancelling the stream can't downgrade an overload/rate-limit to a neutral abandonment (else the breaker never trips).
- **`status = fallback` recording** (#11): the RequestLog records the **served** provider/model + snapshotted price/usage, a sanitized predecessor-failure trail in `routing_reason` (§7.4), and status with precedence — a committed stream that later fails is `error` (not `fallback`), else `fallback` when a predecessor failed, else `success`; a whole-chain failure is one `error` row. `RecordingContext` carries served ids; `RecordOutcome.status` widens to `success|error|fallback`.

## Impact

- Affected specs: **new capability `fallback-routing`**.
- Affected code: additive `chain` in `resolve.ts`; new chain walkers + a signal-aware commit-gate refactor in `ProxyCore`; a small **fix to #6's `withBreakerStream`** (settle-before-yield on error events); `ProxyService` builds the lazy attempt chain (configured order, breaker-wrapped) and records the served model; a `CircuitBreaker` provider in the proxy module (Redis fast-fail + in-memory fallback); `RecordingContext`/`RecordOutcome` served-id + `fallback` status. Reuses #6 breaker/classifiers, #10 commit boundary + drain, #11 recording. No schema migration.
- **Out of scope**: the smart `auto` pipeline (#13/#14); the *optional* first-chunk / first-N-ms buffer that widens the pre-commit window (spec §6.3 "optionally") — deferred as a documented config knob, since commit-on-first-event is the safe default; new provider kinds. Graceful stream drain already ships in #10 (its registry covers the committed stream); the fallback walk respects it. §8's subscription reuse carries a ToS/compliance risk surfaced to users (spec §16) — supported as a kind, not encouraged.
