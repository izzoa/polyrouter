# Design: add-notification-channels (delivery core)

## Context

Reuses: #2 `encryptSecret`/`decryptSecret` + the `PROVIDER_CREDENTIAL_KEY` config pattern; #4 `assertUrlSafe`/`guardedFetch` + the exported IP classifiers (`classifyIp`/`isBlockedIp`) and the allowlist model (relaxes only soft ranges; metadata/loopback never); #1 `registerConfig` + boot fail-fast; the ownership-scoped `OwnedRepository` + global `SessionGuard` (self-restricts to `/api`). New deps: **bullmq**, **nodemailer**. Governing invariants: 6 (SSRF on every server-fetched URL), 8 (encrypt creds, never log secrets — including sanitizing delivery errors), 11 (async, deduped, failure-isolated — never blocks a request/budget path), 5 (tenant isolation). Producers, the scheduler, the breaker read, and password-reset wiring are **#15b**.

## Decision 1 — Data model + at-rest encryption

`notification_channel` (owner-scoped, mirrors #7): `id`, `ownerUserId`/`orgId`, `name`, `kind` (`smtp|apprise`), `enabled`, `encryptedConfig` (text — the whole kind config blob, AES-GCM), `eventsSubscribed` (CSV), `lastTestAt`/`lastTestStatus`, `createdAt`; owner index. `NOTIFY_CREDENTIALS_SECRET` (32-byte hex, gated exactly like `PROVIDER_CREDENTIAL_KEY`) encrypts the config; the API returns a safe view (`kind`, `name`, `enabled`, `eventsSubscribed`, `hasConfig`, `lastTest*`) — never the decrypted blob. Config validated by kind before encrypt:
- `smtp`: `{ host, port, secure: 'none'|'starttls'|'tls', user?, pass?, from, to: string[] }`.
- `apprise`: `{ urls: string[] }`.

`PersistencePort` gains `notificationChannels: OwnedRepository<…>` (owner forced from principal; no unscoped by-id path — invariant 5). The worker loads a channel owner-scoped via `userPrincipal(ownerId)` + the owned repo. **Update semantics** (mirrors #7 + a kind guard): an omitted config **preserves** the encrypted blob, a present config **replaces** it; changing `kind` **requires** a new config (else `422` — an omitted config would leave an incompatible blob); an empty config on an unchanged kind is rejected (a channel must always have deliverable config).

## Decision 2 — SSRF on every notification egress (invariant 6), honest about the boundary

Three surfaces the server (or its sidecar) connects to, each validated to the invariant's standard (**resolved IP at connect time** → rebinding defence). The policy has the **same shape as #4's provider `base_url` guard** (spec §11.2 / invariant 6): **metadata / link-local / all hard ranges are blocked in every mode, never allowlistable**; **loopback** is allowed **only in self-host** (the §11.2 local exception — a loopback SMTP relay / the Apprise sidecar on `127.0.0.1`), blocked in cloud, never allowlistable; a **soft private** range is blocked unless a **port-bounded `NOTIFY_ALLOWED_ENDPOINTS`** entry permits it, in **both** modes (the operator opts their private relay/sidecar in explicitly — no blanket mode relaxation for private ranges). Not the looser "self-host trusts every private address" rule: it stays consistent with the shipped provider guard, and the docker Apprise sidecar (a private compose IP) gets an allowlist entry from #22's compose.

- **`APPRISE_API_URL`** (we POST events to it): its host+port validated via `assertNetworkHostSafe(url.hostname, port, { mode, allowedEndpoints })` at boot. Runs in the **`NotifyRuntime` async factory** that the queue/worker/adapters depend on, so a bad value **fails construction + boot** before a worker exists. A **loopback** sidecar (`http://127.0.0.1:8000`) is allowed in self-host with no extra config; a **private** sidecar (`http://apprise:8000`) needs a `NOTIFY_ALLOWED_ENDPOINTS` entry (documented; #22's compose provides it); a **cloud** deployment pointing at a private/metadata address is refused unless allowlisted. Because `APPRISE_API_URL` is the operator's own trusted, boot-validated sidecar, the delivery POST uses `guardedFetch` with **`providerKind: 'local'`** so the same §11.2 loopback exception applies at connect time (rebinding re-validation retained).
- **User SMTP host** (Nodemailer opens a socket): validated **at connect time in the SMTP adapter** (not merely at CRUD, which a rebind defeats — invariant / spec §11.2). `assertNetworkHostSafe(host, port, { mode, allowedEndpoints })` (shared, reuses #4's `classifyIp`/resolver) resolves the host and applies the policy above (self-host allows a loopback relay; a private relay needs an allowlist entry in either mode; metadata never). Nodemailer then connects to the **pinned validated IP** (`host: <ip>`, `tls: { servername: <originalHost> }` to keep SNI) so the socket cannot rebind to a different address; create-time validation is an early UX reject only.
- **Apprise targets** (the **sidecar**, not us, connects to each): `assertAppriseTargetSafe(rawUrl, ctx)` extracts the **effective** endpoint — for host-bearing schemes (`http`,`https`,`ntfy`,`gotify`,`json`,`jsons`,`form`,`xml`,`matrixs`, **`mailto`/`mailtos`**) the URL host **and** host-override query params (`?smtp=`, `?host=` — `mailto` selects its SMTP server via `smtp=`) → `assertNetworkHostSafe` (same policy); fixed-public-service schemes (`discord`,`tgram`,`slack`,`pover`,`gchat`,…) → allow; unknown → reject (fail-closed). **The authoritative control is network egress policy on the sidecar** (app-side can't defeat rebinding at the container's later connect), so in **cloud** mode Apprise is additionally **gated on `NOTIFY_APPRISE_EGRESS_CONFIRMED=true`** — enforced **at delivery/test-send time** (`deliverApprise` returns `apprise_egress_unconfirmed` in cloud without it), not only at create, so an existing or backup-restored channel can't deliver in cloud without the attestation. Self-host (single-tenant, operator-trusted) allows it with best-effort validation. Documented + carried to #22.

## Decision 3 — Event contract + `emit` (O(1), durable, deduped)

A **structured** discriminated `NotificationEvent` — `{ type, scope, fields }` where `type ∈ {budget_alert, budget_block, provider_down, request_failures_spike, weekly_spend_summary, test}`, and `fields` is a **typed per-type record of operator-facing display metadata** (e.g. `provider_down: { providerName }`, `budget_alert: { limitName, spent, threshold }`) — **no free-form text, no URLs/tokens/raw errors** (a producer contract). The title/body are **rendered in the worker** from `fields` via a fixed template. These display fields are non-secret and legitimately stored in the job (they are not credentials); the secret guarantee is about **channel config**, which is never in the event/job (Decision 5). `scope = { ownerUserId; providerId?; agentId?; limitId?; lifecycleId? }` carries a **lifecycle identity** (`lifecycleId` — the budget period/window, provider incident, or summary week) so distinct lifecycles are not merged by dedup. A pure `scopeKey(event)` yields a canonical collision-safe string over `(type, scope)`; the per-type `windowMs` is **anti-spam rate-limiting within a lifecycle** (e.g. `test` no dedup; `budget_alert` = min re-alert gap keyed on the period; `provider_down` keyed on the incident). #16/#15b supply the lifecycle id.

`NotificationService.emit(event): Promise<void>` — `enqueueFanout` calls `queue.add('fanout', { event }, { deduplication: { id: hash(type + scopeKey), ttl: windowMs }, ...opts })` on the **producer** connection, wrapped in a `withDeadline(2s)` so a blackholed Redis can't stall the caller (see Decision 4). BullMQ's **TTL-based deduplication** drops a duplicate within `ttl` **independently of job retention** (a plain `jobId` would stop deduping once `removeOnComplete` reaps it, and can't contain `:`) — durable, accept-at-most-once per window, and O(1) (no DB query on the emit path). The whole call is wrapped in try/catch so a Redis-down / deadline reject is **caught + logged (opaque), never blocking or thrown into the caller** (invariant 11).

## Decision 4 — BullMQ: two connections, two job types, isolated retry

- **Connections**: producer = `REDIS_CLIENT.duplicate({ maxRetriesPerRequest: 3, connectTimeout: 2000, commandTimeout: 2000 })`, worker = `REDIS_CLIENT.duplicate({ maxRetriesPerRequest: null })` (BullMQ-required); both eagerly connected (the base is `lazyConnect`) and closed on `OnApplicationShutdown`. **Bounded emit**: `enqueueFanout` wraps `queue.add` in a `withDeadline(2000ms)` that rejects if the add doesn't settle — the connection-level `commandTimeout`/`enableOfflineQueue:false` alone do **not** bound BullMQ's `waitUntilReady` (proven by a real blackhole test, not a mocked reject), and `enableOfflineQueue:false` would also drop emits during the brief post-boot connect window, so the **offline queue is left enabled** and `withDeadline` is the sole bound. **Robustness**: the `Queue` and `Worker` each get an `'error'` listener (an unhandled EventEmitter `'error'` on a down Redis would crash the process); `OnApplicationShutdown` bounds `worker.close()`/`queue.close()` with a deadline then force-disconnects, so a worker blocked on an unreachable Redis can't hang shutdown (invariant 12). A `whenReady()` awaits producer+worker readiness for callers that must not race startup.
- **Fan-out job** (`type:'fanout'`): loads the owner's `enabled` channels subscribed to `event.type` (DB query here, **off** the emit path) and enqueues one **delivery** job per channel with an **opaque** `jobId` (`hash(type,scopeKey,channelId, windowBucket)`, no `:`), where `windowBucket` derives from the **fan-out job's own immutable timestamp** (not retry time) so a fan-out retry crossing a window boundary can't mint fresh delivery ids.
- **Delivery job** (`type:'deliver'`, `{ channelId, ownerId, event }` — the **structured event only, no secret**): loads the channel (owner-scoped) → skip if missing/disabled → decrypt config → **render** the title/body from `event.fields` → dispatch to the SMTP/Apprise adapter under a per-send timeout. Job opts: `attempts: 4`, exponential backoff, `removeOnComplete: { age: 3600 }`, `removeOnFail: { age: 86400 }` (**age-based**). A delivery that fails all attempts throws a **sanitized code** (Decision 5) → BullMQ marks it failed and **moves on** — isolated from other channels/events. Worker `concurrency` bounded (default 5).

Delivery semantics are **at-most-once accepted per window** (jobId dedup) + **best-effort idempotent send** — BullMQ is at-least-once (a stalled job can re-run), and exact-once external SMTP is not guaranteeable; the delta states this honestly.

## Decision 5 — Sanitized errors (invariant 8)

The adapters catch the raw Nodemailer/undici error and rethrow a **fixed code** — `smtp_send_failed`, `apprise_unreachable`, `apprise_http_5xx`, `apprise_not_configured`, etc. — with **no** host/recipient/URL/response text. So BullMQ's `failedReason`, every log line (which records only the code + opaque job id, never the job object), `last_test_status`, and the API responses carry only the code. Combined with the **config-free job** (only `channelId`/`ownerId`/`event`, where `event.fields` are non-secret display metadata rendered in the worker — Decision 3), there is no path for a **channel credential** (SMTP pass, token-bearing URL) or a raw error (host/recipient/response) into Redis, logs, or the API. A canary test asserts a secret sentinel placed in a channel's **config** never appears in the Redis job record, the DB status, the API response, or captured logs. (Display fields — a provider name, a limit name — are non-secret by the producer contract and legitimately appear in the queued job.)

## Decision 6 — Delivery adapters

- **SMTP** (`nodemailer`): **connect-time** — `assertNetworkHostSafe(host, port)` → transport pinned to the validated IP (`host: <ip>`, `tls: { servername: <host> }` for SNI), `secure: mode==='tls'`, `requireTLS: mode==='starttls'`, `ignoreTLS: mode==='none'`, `auth` when user/pass present, `connectionTimeout`/`greetingTimeout`/`socketTimeout` set; `sendMail({ from, to, subject, text })` (subject/text rendered from `event.fields`); `transport.close()` in `finally`.
- **Apprise**: `guardedFetch(joinUrl(APPRISE_API_URL, '/notify'), { method:'POST', headers:{'content-type':'application/json'}, body: JSON.stringify({ urls, title, body }), signal: AbortSignal.timeout(sendTimeoutMs) }, { context:{ mode }, allowedEndpoints })`; check `response.ok` (else `apprise_http_<class>`); **consume/cancel** the body under the timeout on every path. `APPRISE_API_URL` unset → `apprise_not_configured`.

## Decision 7 — Test-send

`POST /api/notification-channels/:id/test`: load the channel (scoped) → deliver a `test` event **directly** (bypass the queue, under a timeout) so the user gets **inline** success/failure → persist `lastTestAt` + `lastTestStatus` (`success` or `failed:<code>`) → return `{ ok, error? }` (sanitized). Direct is correct for a deliberate synchronous action; event delivery stays async.

## Decision 8 — Degradation & resilience (invariant 11)

- `emit` never blocks or throws: producer connection is fail-fast (`enableOfflineQueue:false`) + a bounded `queue.add` wrapped in try/catch.
- A failing channel is a failed BullMQ job in isolation; other channels/events proceed.
- Redis down → `emit` logs + returns (event dropped, non-blocking); the request/budget path is untouched. Delivery durability that matters (once accepted) is BullMQ's; acceptance under Redis-down is best-effort by design.
- All secrets encrypted at rest, decrypted only in the worker/test-send, and errors sanitized — nothing sensitive in logs, the job store, or the API.

## Risks / trade-offs

- **Apprise sidecar egress** — even with connect-time host+override validation, the sidecar makes the real per-target connect; the authoritative control is network egress policy on the container (deployment/#22). Cloud Apprise is therefore **gated** behind `NOTIFY_APPRISE_EGRESS_CONFIRMED`; self-host allows it (operator-trusted). Stated, not hidden.
- **Host policy = #4's provider shape** — self-host allows a **loopback** relay/sidecar out of the box (the §11.2 local exception); a **private** relay/sidecar (incl. the docker `apprise:8000`) needs a port-bounded `NOTIFY_ALLOWED_ENDPOINTS` entry in **either** mode (#22's compose provides it); metadata/link-local is refused in every mode; cloud additionally blocks loopback. No looser "self-host trusts every private address" rule — it stays consistent with the shipped provider guard. The adapter always pins the validated IP so a rebind cannot redirect the socket, in either mode.
- **At-least-once delivery** — a duplicate send is possible (BullMQ stalled-job re-run); acceptable for alerts, stated in the delta. Dedup is TTL-based (`deduplication:{id,ttl}`), independent of job retention.
- **Two new deps** — bullmq + nodemailer, the spec's named tools (§3.1/§10.1).
