# Add notification channels — the delivery core (SMTP/Apprise, async + resilient)

## Why

The router needs a way to deliver alerts (a tripped provider, a budget threshold, a failure spike) that **decouples what triggers a notification from how it's delivered** (spec §10.1) and runs **off the request path** so a slow or dead channel never stalls a request or a budget check (invariant 11). This change lands the **delivery core**: the `NotificationChannel` model + CRUD, credential encryption, SSRF validation of every server-fetched notification URL, the SMTP and Apprise delivery adapters, a durable BullMQ delivery pipeline with dedup, and per-channel test-send. It exposes a `NotificationService.emit(event)` API that #16 (budget alerts) and #15b (`provider_down`, failure-spike, weekly summary, password-reset) build on.

Scope split (this is #15a of §15; #15b adds the producers): #15's §15 acceptance spans channels + delivery (here) and event producers (`provider_down`, password-reset — #15b). This change satisfies the delivery/channel portion; #15b completes the producers.

Non-negotiables addressed here: channel credentials **encrypted at rest** and never logged, and delivery errors **sanitized** so no host/recipient/URL/secret reaches a log or the BullMQ job store (invariant 8); every server-fetched notification URL — user SMTP hosts, Apprise targets, and `APPRISE_API_URL` — **SSRF-validated** (invariant 6); delivery **async, deduplicated, and failure-isolated** (invariant 11); channels **tenant-scoped** (invariant 5).

## What Changes

- **`NotificationChannel` table + migration** (`shared`): owner-scoped `notification_channel` (kind `smtp|apprise`, `enabled`, `encrypted_config`, `events_subscribed`, `last_test_at`/`last_test_status`) + an `OwnedRepository` accessor. **Config encrypted at rest** with a new `NOTIFY_CREDENTIALS_SECRET` (mirrors #7); the API returns a safe view (never the secret).
- **Channel CRUD** (`/api/notification-channels`, session-guarded): list/create/get/update/delete + **test-send**, class-validated DTOs, ownership-scoped service. Create/update validate + **SSRF-check** the config; test-send delivers a sample event **directly** (bypassing the queue, under a timeout) and persists a **sanitized** `last_test_status`/`at` for inline feedback.
- **SSRF for every notification egress** (`shared`, reuses #4), validated to the invariant's **connect-time resolved-IP** standard:
  - `APPRISE_API_URL` (we POST to it) — its host validated **at boot, before the queue/worker construct** (an async `NotifyRuntime` factory throws on failure) under **#4's provider-guard policy**: a **loopback** sidecar (`http://127.0.0.1:8000`) is allowed in self-host; a **private** sidecar (`http://apprise:8000`) needs a port-bounded `NOTIFY_ALLOWED_ENDPOINTS` entry (#22 provides it); cloud refuses private/metadata unless allowlisted; metadata never. The POST uses `guardedFetch` with `providerKind:'local'` (the operator's own boot-validated sidecar) so the same loopback exception applies at connect time.
  - **User SMTP host** (Nodemailer connects) — `assertNetworkHostSafe(host, port)` validated **in the adapter at send**, and Nodemailer connects to the **pinned validated IP** (SNI preserved) so a rebind can't redirect the socket. Same policy shape as #4: self-host allows a **loopback** relay (the §11.2 local exception); a **private** relay needs a port-bounded allowlist in either mode; metadata never.
  - **Apprise targets** — a scheme-aware check validating the URL host **and** host-override params (`?smtp=`/`?host=` — `mailto` selects its SMTP server that way); fixed-public-service schemes allowed, unknown rejected. Because the Apprise *sidecar* makes the real per-target connect, **cloud** Apprise is **gated behind `NOTIFY_APPRISE_EGRESS_CONFIRMED`** (operator attests the container is network-isolated); self-host allows it. The authoritative control is network egress policy on the sidecar (documented, carried to #22).
- **Durable async delivery via BullMQ** (`control-plane`, new dep) on **two** dedicated ioredis connections: a **producer** and a **worker** (`maxRetriesPerRequest:null`). `emit(event)` enqueues **one** fan-out job via `enqueueFanout`, whose `queue.add` is wrapped in a **`withDeadline(2s)`** so `emit` is bounded even against a blackholed Redis and never blocks (the connection-level `commandTimeout` alone does not bound BullMQ's `waitUntilReady`, and disabling the offline queue would drop emits in the brief post-boot connect window — so the deadline wrapper is the bound). Fan-out uses BullMQ **TTL-based deduplication** (`deduplication:{ id: hash(type,scopeKey), ttl: windowMs }` — durable and independent of job retention); it loads the owner's subscribed channels and enqueues **per-channel** delivery jobs (opaque ids), so a failing channel is retried with bounded backoff then **logged + dropped in isolation** — never stalling other channels, events, the request path, or budget enforcement (invariant 11). `Queue`/`Worker` carry `'error'` listeners and shutdown is deadline-bounded (a down Redis can't crash the process or hang shutdown).
- **Structured, secret-free events + sanitized delivery**: a discriminated `NotificationEvent` carries typed non-secret **fields** (no free-form text/URLs) that the **worker renders** into title/body, and the queued job holds only `channelId`/`ownerId`/`event` (no config). **SMTP** via Nodemailer (TLS mode → `secure`/`requireTLS`/`ignoreTLS`, timeouts, transport closed); **Apprise** via `guardedFetch` to `<APPRISE_API_URL>/notify` (`RequestInit` + `AbortSignal`, `response.ok` checked, body consumed). Both throw only **fixed sanitized codes** — no host/recipient/URL reaches BullMQ `failedReason`, logs, `last_test_status`, or the API. Failed jobs use **age-based** retention.
- **Event scope contract**: a canonical per-type `scopeKey` (owner + optional `providerId`/`agentId`/`limitId`) + a service-owned per-type window, so #16/#15b call `emit` without re-deriving dedup identity.
- **Config** (`notify.config.ts`, in the #1 fail-fast schema): `APPRISE_API_URL`, `NOTIFY_CREDENTIALS_SECRET`, `NOTIFY_ALLOWED_ENDPOINTS`, `NOTIFY_APPRISE_EGRESS_CONFIRMED`. (Server-wide `SMTP_*` defaults are #15b's — a channel carries its own SMTP config.)

## Impact

- Affected specs: **new capability `notification-channels`**.
- Affected code: `notification_channel` table + migration + persistence accessor (`shared`, `control-plane/database`); `assertNetworkHostSafe` + `assertAppriseTargetSafe` SSRF helpers (`shared`); new `control-plane/src/notifications/` (service + emit, BullMQ queue/worker/fan-out, SMTP/Apprise adapters, CRUD, `notify.config.ts`). New deps: **bullmq**, **nodemailer** (+ `@types/nodemailer`). Reuses #2 encryption, #4 SSRF, Redis wiring.
- **Out of scope → #15b**: event producers (`provider_down` via a breaker transition event, `request_failures_spike`, `weekly_spend_summary`), the interval/leader scheduler, the unscoped provider→owner projection, the RequestLog time/status query, the breaker read, and the Better-Auth password-reset wiring.
- **Out of scope (deferred elsewhere)**: budget `budget_alert`/`budget_block` **producers** (#16 — the event types + subscription exist here); the Settings → Notifications **UI** (#20); the Apprise **CLI fallback** (API mode is the recommended default, §10.1). Delivery semantics are **at-most-once accepted per window + best-effort idempotent send** (BullMQ is at-least-once; exact-once external SMTP delivery is not guaranteed).
