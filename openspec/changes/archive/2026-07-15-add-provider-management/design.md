# Design: add-provider-management

## Context

`Provider`/`Model` tables and their tenant-scoped repositories already exist (#2's `PersistencePort.providers` / `.models`); `encryptSecret`/`decryptSecret`, `assertUrlSafe`, and `createProviderAdapter` are shipped (#2/#4/#6). This change is the session-authenticated management surface that ties them together: Provider CRUD + `test-connection`/`sync-models`, credential encryption at rest, CRUD-time SSRF validation, and a Models list/filter API. It follows the existing `agents.controller` conventions (global `SessionGuard`, `@CurrentPrincipal()`, `PERSISTENCE_PORT`, `class-validator` DTOs, `toSafe` mappers).

## Goals / Non-Goals

**Goals:** tenant-scoped Provider CRUD with encrypted-at-rest credentials never returned or logged; every server-fetched `base_url` SSRF-validated at write time and before each outbound action, with the custom/local endpoint never allow-listed; model-catalog sync (ids + display names + `last_synced_at`, no prices) via #6's adapter; a Models list/filter API; a `PROVIDER_CREDENTIAL_KEY` config with dev-fallback gating.

**Non-Goals:** pricing/`ModelPrice`/`refresh-pricing` (#8), routing config/proxy (#9/#10), any change to #4's SSRF ranges or #6's adapter behavior, and any schema migration (the tables exist).

## Decisions

1. **A `ProvidersModule` with two controllers + a service.** `ProvidersController` (`api/providers`), `ModelsController` (`api/models`), and `ProvidersService` (encryption, SSRF gate, adapter construction, model upsert). Registered in `app.module`; the global `SessionGuard` covers `/api` automatically, and `PERSISTENCE_PORT` resolves through the global `DatabaseModule`. This keeps the controllers thin and the secret-handling in one audited place.

2. **Credentials are encrypted on write, decrypted only in-memory, never surfaced (invariant 8).** `create`/`update` call `encryptSecret(credential, key)` and store the envelope in `provider.encrypted_credentials`; the plaintext exists only for the duration of the request. `test-connection`/`sync-models` `decryptSecret` it solely to construct the #6 adapter. The API's `toSafe(provider)` returns `{ id, name, kind, protocol, baseUrl, status, hasCredential, createdAt }` — **never `encrypted_credentials` and never the plaintext** (and no `lastSyncedAt` — that column is on `Model`, not `Provider`, Round-1 Medium-4). No log line includes either. A test asserts the credential is absent from every response body and that the stored column is a `poly-enc:` envelope, not the plaintext.

3. **The encryption key is `PROVIDER_CREDENTIAL_KEY`, resolved like the auth secrets.** A `providers` config fragment registers `PROVIDER_CREDENTIAL_KEY` (32-byte hex, optional). `resolveCredentialKey(cfg, base)` mirrors `resolveAuthSecrets`: a fixed dev-fallback key is permitted **only** on a loopback-bound, non-production, self-hosted instance; anything network-reachable or production requires a real key (value never echoed in errors). This matches the established secret-gating policy rather than inventing a new one.

4. **Every server-fetched `base_url` is SSRF-validated at write time AND before each action, and never carries embedded credentials (invariant 6, Round-1 High-3).** `create`/`update` and the top of `test-connection`/`sync-models` call `assertUrlSafe(baseUrl, { context: { mode: MODE, providerKind: kind } })`. A blocked address → a `422` (validation error), except the `kind:'local'` + `MODE=selfhosted` loopback exception (from #4). Because `assertUrlSafe` does not reject URL **userinfo**, the service SHALL additionally reject any `base_url` with a non-empty `username`/`password` (a `https://user:token@host` would otherwise store a credential in the plaintext `base_url` column and every response) and reject a query/fragment (the adapter joins paths by string concatenation, so `base_url` must be an origin + optional path prefix). The **normalized** URL (from the parsed `URL`) is persisted. The `base_url` is a free field — there is **no closed provider allow-list** (spec §8). #6's adapter re-validates at connect time (DNS-rebinding defense); the CRUD-time gate gives an immediate, clear rejection and covers the write path even before any request is made. `kind:'local'` under `MODE=cloud` is rejected outright (mirroring #6's factory).

5. **Actions build the #6 adapter through an injectable factory, and their errors are sanitized (Round-1 High-2, Round-2 High-1).** `ProvidersService` depends on a `PROVIDER_ADAPTER_FACTORY` token (default `createProviderAdapter`) so unit tests inject a fake adapter and never hit the network. `buildAdapterConfig(provider)` requires a non-null `baseUrl` (guaranteed by create; a missing one is a clear `422`, not an adapter call). A credential is **required for `api_key`/`subscription`/`custom`** (a missing one is a clear `422` before any adapter call); only `local` may omit it, in which case an empty string is passed (local servers ignore the auth header) — so `credential = provider hasCredential ? decrypt(envelope) : ''`. **The service never returns or logs the adapter's raw `ConnectionResult.message`, a thrown error, or the `config`** — #6's classifier can embed up to 200 bytes of an upstream error body, so every failure is mapped to a **fixed public message keyed on `{ kind, status }`**. Allowlisted log metadata is `{ kind, status, traceId }` where **`traceId` is generated internally by this service** — it never logs #6's `ProviderError.requestId` (that comes from an upstream-controlled `x-request-id` header a hostile endpoint could set to the credential). `test-connection` maps the sanitized result → `provider.status` (`ok`/`error`); `sync-models` catches and sanitizes adapter throws the same way.

6. **Model sync is an atomic, provider-scoped upsert; prices are #8's (Round-1 High-1).** The `list-then-create/update` dance races (two concurrent syncs both see an absent model → a `UNIQUE(provider_id, external_model_id)` violation) and can't be done safely with the current `ModelAccessor` (its `listForPrincipal` isn't provider-scoped and `update` keys on tenant, not provider). So this change **adds `ModelAccessor.upsertForProvider(principal, providerId, values)`** — an ownership-checked `INSERT … ON CONFLICT (provider_id, external_model_id) DO UPDATE SET display_name, last_synced_at` (returns `null` if the provider isn't owned by the principal). No schema migration (the unique index exists from #2); this is a persistence-port method addition. `sync-models` **deduplicates** `listModels()` by id, then upserts each — writing **ids, display names, and `last_synced_at` only, no prices/capability flags** (null/false for #8). Models absent from the latest sync are **left in place** (they may be referenced by tiers — pruning is a later deliberate concern; the change reports the synced count rather than silently deleting). Cross-tenant parenting fails closed inside the statement.

7. **Local providers are gated and marked free downstream.** `kind:'local'` requires `MODE=selfhosted` (rejected otherwise, `422`). Marking local/curated models `is_free` is #8's catalog concern, not this layer — `sync-models` does not set `is_free`.

8. **DTOs validate the two enums and the URL; everything else is free.** `CreateProviderDto`: `name`, `kind ∈ {api_key,subscription,custom,local}`, `protocol ∈ {openai_compatible,anthropic_compatible}`, `baseUrl` (a valid http(s) URL — the SSRF gate + userinfo/query rejection do the rest), optional `credential`. `UpdateProviderDto`: partial. Global `ValidationPipe` (`whitelist`, `forbidNonWhitelisted`) rejects unknown fields. The `ModelsController` list accepts `providerId?`, `isFree?`, and capability filters as query params.

9. **Update validates the EFFECTIVE merged config and preserves the credential unless supplied (Round-1 Medium-5).** A partial PATCH is validated against the *resulting* row, not the patch alone: the service fetches the owned provider, computes `nextKind = patch.kind ?? stored.kind` and `nextBaseUrl = patch.baseUrl ?? stored.baseUrl`, and SSRF-gates that tuple (so flipping `local`→`custom` without a new `baseUrl`, or changing only a `local` provider's URL, validates correctly) before persisting — an invalid unusable row is never stored. The credential is re-encrypted **only when the `credential` field is actually present** in the patch; omitting it preserves the existing envelope, and an explicit empty string clears it (`encrypted_credentials = null`, `hasCredential:false`).

10. **Provider delete cascades to models and their routing entries — intentionally and tested (Round-1 Medium-6).** The schema's `ON DELETE CASCADE` (provider→model→routing_entry) means deleting a provider removes its models and any tier assignments that referenced them. This is the correct semantics (you cannot route to a deleted provider's models), but it is destructive, so it is stated explicitly and covered by a test (delete a provider → its `Model` rows and their `routing_entries` are gone), rather than left as an accidental side effect.

## Risks / Trade-offs

- [Credential leakage — three vectors] → (a) encrypted at rest, decrypted only in-memory, stripped from responses by `toSafe`, never logged; (b) action errors mapped to fixed `{kind,status}` messages so a hostile endpoint reflecting the auth header can't surface it; (c) URL userinfo rejected so a credential can't hide in `base_url`. A test drives all three (fake adapter echoing the exact credential; a `user:pass@` URL; a grep of responses + DB column + captured logs).
- [SSRF on a user base_url] → validated at write time and before each action with the per-kind context, again at connect time by #6; local loopback only under self-host; userinfo/query rejected; normalized URL persisted.
- [Model sync race / duplicate ids] → an atomic `ON CONFLICT` upsert keyed on `(provider_id, external_model_id)` with in-statement ownership; the service dedupes adapter results; sync never prunes.
- [PATCH validating only the delta] → the effective merged `(nextKind, nextBaseUrl)` is validated; the credential is preserved unless explicitly supplied.
- [A real network call in an action] → the adapter factory is injectable, so unit tests are network-free; one e2e uses the real factory against a loopback stub to prove default wiring.
- [New env var] → `PROVIDER_CREDENTIAL_KEY` reuses the exact dev-fallback gating of the auth secrets, so self-host dev stays zero-config while cloud/prod requires a real key.

## Migration Plan

Additive: a new control-plane module + a config fragment + **one new `ModelAccessor.upsertForProvider` method on the persistence port** (interface in `@polyrouter/shared/server`, impl in the control-plane database port). **No schema migration** — the `provider`/`model` tables, the `UNIQUE(provider_id, external_model_id)` index, and the tenant-scoped repos all exist from #2; the upsert uses that existing index. Rollback = unregister the module/config and remove the port method.

## Open Questions

None blocking. Whether `test-connection` should also do a minimal `chat` probe for providers lacking a `/models` endpoint is deferred — `listModels()` is the cheap default; #6's `testConnection` already encapsulates the choice, and a provider-specific fallback can be added there without touching this layer.
