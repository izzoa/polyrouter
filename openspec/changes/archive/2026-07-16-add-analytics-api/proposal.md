# Proposal: add-analytics-api

## Why

The dashboard's **Observe** half (spec §2 step 4, §9) — Overview charts, the Costs breakdowns, and the Requests log — has no data source yet: #18 wired the management pages but left Overview/Requests/Costs on the simulator ("preview — simulated") precisely because these aggregation endpoints don't exist. This change adds the control-plane `analytics/` module (§9): time-bucketed `GROUP BY` reads over the immutable RequestLog (#11) + its cascade-attempt cost ledger (#14), all **tenant-scoped** (invariant 5), on plain Postgres (Timescale/ClickHouse are deferred cloud graduations, §3.3). It unblocks #19 (the analytics UI). No schema change — the `(owner_user_id, created_at)` composite index added in #16 already serves the range scans.

## What Changes

- **A tenant-scoped `analytics` accessor on the `PersistencePort`** (the central owner-scoped seam — every query carries `ownershipPredicate`, no unscoped-by-owner fetch; invariant 5). SQL lives in a `control-plane/database/analytics.queries.ts` helper to keep `port.ts` lean. Four reads over `request_log` (+ the `request_attempt` cascade ledger for spend):
  - `summary(range)` — totals for the Overview stat cards: `spend`, `requests`, input/output/cache tokens, `successCount`/`fallbackCount`/`errorCount`, `escalatedCount`, `estimatedCount`, and a free/paid/unpriced request split.
  - `timeseries(range, bucket)` — `date_trunc`-bucketed (`hour`/`day`/`week`/`month`) points `{ bucket, requests, spend, inputTokens, outputTokens, errorCount, fallbackCount, escalatedCount }` for the messages-over-time / spend charts (sparse — only non-empty buckets; the UI fills gaps).
  - `breakdown(range, dimension, limit)` — top-N by spend for `model`/`provider`/`agent`/`tier`: `{ key, label, spend, requests }` (the Costs bars + Overview "top models"). `label` is resolved via a LEFT JOIN to the owned table (`externalModelId`/provider `name`/agent `name`; tier key is already human-readable), null if the row was since deleted.
  - `listRequests(query)` — **keyset-paginated** (`created_at`,`id` DESC, PK tie-break) request-log listing with `decisionLayer` + `routingReason` (the Requests table + decision inspector, §9's transparency feature) plus owner-scoped model/provider/agent labels and this request's attempt cost, filterable by `status`, `decisionLayer`, and `escalated`.
- **Spend includes both ledgers, µ$-rounded to match budgets.** Every spend figure sums `request_log.cost` + `request_attempt.cost` (the superseded cheap cascade call) with the **same per-row integer micro-dollar rounding #16's budget reader uses**, so dashboard spend reconciles with the budget a user set (a float sum would diverge by cents). Request counts/tokens/free-paid split are over served `request_log` rows; only spend adds attempts. Costs come straight from the immutable per-request snapshots (invariant 4); analytics never re-prices against the current catalog. `model`/`provider`/`tier` breakdowns union both ledgers by their own id/key; the `agent` breakdown attributes attempts via their parent log (attempts carry no `agent_id`). Unpriced (null-cost) rows count toward request/token totals but not spend.
- **A session-guarded `AnalyticsController` at `/api/analytics`** — `GET /summary`, `/timeseries`, `/breakdown`, `/requests` — each taking a validated ISO `from`/`to` range (class-validated DTOs; a bounded max window; `from < to`), `@CurrentPrincipal()`-scoped, returning safe views (never `ownerUserId`/`orgId`). Registered in `app.module`.

Governing invariants: **5** (every read owner-scoped through the shared port; no by-owner-less fetch, no cross-tenant leakage), **4** (spend is the immutable snapshot cost, never recomputed), **9** (no tokenizer/generative call on these read paths — plain SQL aggregates). Out of scope (deferred): the analytics UI itself (#19), a subscription-vs-API cost sub-split (needs provider-kind denormalized onto the log), contiguous zero-filled buckets (the UI fills), and any Timescale/ClickHouse graduation (§3.3).
