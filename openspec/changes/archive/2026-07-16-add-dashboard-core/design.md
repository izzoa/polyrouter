# Design: add-dashboard-core

## Context

Reuses: the whole built SPA (`packages/frontend`, SolidJS + Vite, served on one port by `configureSpa`, `/api`+`/v1` Vite-proxied to `:3001` in dev, dev CORS pinned to `DASHBOARD_ORIGIN` with credentials); #3's Better Auth (`/api/auth/*`, `SessionGuard` global on `/api`, localhost auto-login, first-user-admin, conditional google/github/discord OAuth); #3's Agents API (`/api/agents` create/rotate return `key`+`snippet` **once**); #7's Providers/Models API (`/api/providers` + `test-connection`/`sync-models`, `/api/models`); #9's Routing API (`GET /api/routing/tiers`, `PUT /api/routing/tiers/:id/entries` — atomic ordered chain, position 0 = primary); #8's pricing (`/api/pricing`, `resolveModelPrice` precedence: **model-own price → catalog → null**); the shared `connectionSnippet`/`HarnessType`. Governing invariants: **5** (owner-scoped everything), **4** (immutable snapshot cost), **8** (write-only/shown-once secrets), **6** (SSRF, unchanged).

**The seam.** The prototype was deliberately built with the data boundary at the store↔`src/data/*` layer (comments in `simulator.ts`/`seed.ts`/`catalog.ts` say so). So the swap replaces *how `app.*` actions get/mutate data* — real `fetch` instead of in-memory — while pages/components keep reading `state.*` and calling `app.*`. Only the management slice (auth, agents, providers, onboarding, account) is realized here; the observe/config pages keep the simulator.

## Decision 1 — Backend: three dashboard endpoints (owner-scoped, no deps)

- **`GET /api/me`** (session-guarded, in a new small `account` module). `@CurrentPrincipal()` gives `{ userId }` under **both** cookie auth and auto-login (the guard sets it either way). Resolve identity via a new `IdentityPort.getIdentity(userId): { id, email, name, role } | null` (a plain `users` select added to `buildIdentityPort`), and return `{ userId, email, name, role, mode }`. This is the SPA's authorization probe: 200 = authorized (self-host loopback lands here via auto-login, with **no cookie** — which is exactly why `get-session` is insufficient and `/api/me` is needed); 401 = show login.
- **`GET /api/login-config`** (`@Public()`, same module). **Named to avoid the `/api/auth*` prefix:** `auth/mount.ts` intercepts every `req.path.startsWith('/api/auth')` on the raw Express instance *before* Nest routing, so a `/api/auth-config` route would be swallowed by Better Auth (404). Returns `{ mode, emailPassword: true, oauthProviders: ('google'|'github'|'discord')[] }` from a pure `enabledOauthProviders(authConfig)` resolver (a provider is listed iff both its `*_CLIENT_ID` and `*_CLIENT_SECRET` are set — mirrors better-auth's own conditional wiring). Drives the login page's buttons; no secrets. Its e2e MUST boot through the real `mountAuth` ordering (not a controller-only harness) so the prefix collision can't regress.
- **`PATCH /api/models/:id`** (session-guarded; add to the existing `models.controller`; the method lives on the already-injected `ProvidersService`). **custom/local only:** the service resolves the model AND its owner-scoped provider and rejects (422) any kind other than `custom`/`local` — model-own price is the *top* of `resolveModelPrice`'s precedence, so allowing it on `api_key`/`subscription` would silently bypass the bundled catalog (the correct path for known providers). Routed through the **existing owner-scoped `ModelAccessor.update`** (models owned through their provider — no unguarded by-id, invariant 5; cross-tenant by-id is 404). **Atomic pricing semantics** (because `resolveModelPrice` applies a model-own price only when *both* prices are non-null, and `computeCost` returns 0 when `isFree`): validate the **fields present in the request body** (not merged with the existing row) — exactly one of *"free"* `{ isFree:true }` → `{ 0, 0, isFree:true }`, or a *price pair* `{ inputPricePer1m, outputPricePer1m }` (both present together) → `{ …, isFree:false }`; a lone price (even onto an already-priced model), `isFree:true` with non-zero prices, or an empty body is 422. Prices validate with `@IsNumber({ allowNaN:false, allowInfinity:false })` + `@Min(0)`. `SafeModel` gains `inputPricePer1m`/`outputPricePer1m` so the view reads them. Not effective-dated and does not touch RequestLog — the recorder copies prices into the snapshot at completion (invariant 4); only future requests see the new price. The bundled-catalog effective-dated override for *known* models already exists (`POST /api/pricing/:modelKey/override`) and is surfaced read-only.

## Decision 2 — Frontend: the API client seam (injectable, test-stubbable)

A single `src/data/api.ts` (`ApiClient`) of typed methods over `fetch` with `credentials: 'include'`, **relative** `/api/...` paths (same-origin prod, Vite-proxied dev — never the display-only `BASE_URL`, which is `/v1` for snippets), a JSON body helper, and a normalized error (`{ status, code, message }` parsed from the Nest error envelope). `createAppStore(client?: ApiClient)` takes an optional client (defaults to the real one); **Vitest injects a fake** — this is the new test seam (no fetch mocking exists today; the existing `createAppStore()`/`live={false}` patterns extend cleanly). Actions that were synchronous simulator mutations become `async` calls that set a per-slice `loading`/`error` and then patch the store from the response.

## Decision 3 — Auth bootstrap + gate

App root gains a bootstrap: `await client.me()` →
- **ok** → `state.session = { userId, email, role, mode }`, render the shell (self-host loopback: immediate, as admin).
- **401** → `await client.loginConfig()` → render **`Login`** (a new gate view, not one of the sidebar `Page`s): email+password **sign-up (requires `name` + email + password — better-auth's `signUpEmail` needs `name`)** and sign-in (`POST /api/auth/sign-up|sign-in/email`, then re-run bootstrap), and one button per `oauthProviders` entry. **OAuth start:** `POST /api/auth/sign-in/social` returns `{ url }` — the client MUST navigate to that `url` (a bare POST does not begin the flow); `callbackURL` is a fixed SPA path (safe — better-auth restricts it to `trustedOrigins`, no open redirect). **Dev-origin caveat:** OAuth callback cookies are host-only, so `BETTER_AUTH_URL` (default `127.0.0.1:3001`) and `DASHBOARD_ORIGIN` (default `localhost:3000`) must be unified for a dev OAuth round-trip (document: set `BETTER_AUTH_URL` to the dashboard host, or run Vite on `127.0.0.1`); email/password + auto-login are unaffected.
- A **logout** action calls `POST /api/auth/sign-out` then **re-runs the bootstrap** (`client.me()`): under loopback auto-login `me` stays 200, so it remains `ready` and shows an "auto-login active — sign-out is inert here" note rather than a false gate; with a real cookie, `me` becomes 401 and the gate shows.

The Settings account block reads `state.session` (real email/role/mode) instead of the hardcoded `admin@localhost`; admin-only affordances gate on `role === 'admin' && mode === 'selfhosted'`. Bootstrap failures that aren't 401 (network/500) render a retryable error, never a blank app or a false login prompt.

## Decision 4 — Agents / Providers / Onboarding realization

- **Agents:** `list()` → `GET /api/agents` (`SafeAgent[]`: id/name/harness/prefix/lastUsedAt/createdAt); `create({name,harness})`/`rotate(id)` → the key-reveal modal shows the **server-returned `key`+`snippet`** (shown once). The key lives only in transient store state, is **never persisted** (no `localStorage`), and is **cleared on modal dismissal / sign-out / error**. The current per-row "Snippet" reveal is **removed** — a working snippet can't be reconstructed without the raw key; re-obtaining one is a rotate. Add a **delete** action (`DELETE /api/agents/:id`, confirm). Per-agent request/spend columns need #17 → render "—".
- **Providers:** map UI kind `api|sub|custom|local` ⇄ API `api_key|subscription|custom|local`. Reconcile the flow to the backend's **create-then-act**: `POST /api/providers` (**a full form: name + kind + protocol + baseUrl + credential**) → then `POST /:id/test-connection` and `POST /:id/sync-models` (both return the sanitized `ActionResult` at HTTP 200 even on failure — branch on `result.ok`, not the HTTP status — and update `status`). Model list per provider via `GET /api/models?providerId=`; **custom/local price editor** via `PATCH /api/models/:id` (local defaults to free). **Subscription ToS UX:** on `kind==='subscription'`, show the §16 risk note and a nudge/CTA to add a pay-per-token fallback provider. Credentials are input-only; the card shows `hasCredential`, never a value. Provider cards show only the real `status` (`unknown`/`ok`/`error`); the prototype's "circuit closed/half-open" copy is **removed** (live breaker health is #21).
- **Onboarding:** persist for real, with an explicit **failure-aware state machine** (each step stops and surfaces a clear error on failure) — Step 1 `POST /api/agents`; Step 2 a **full provider form** (name/kind/protocol/baseUrl/credential, reusing the Providers add-form) → `POST /api/providers` → `POST /:id/sync-models` (stop if `result.ok===false` or `synced===0`) → `GET /api/models?providerId=` → `PUT /api/routing/tiers/:defaultId/entries` with `{ modelIds:[firstId] }` (default tier from `GET /api/routing/tiers` where `key==='default'`; stop if absent); Step 3 **verify**: POST `{model:'auto', messages:[…]}` to `/v1/chat/completions` with `Authorization: Bearer <mintedKey>` and render the real response (agent-key plane, no cookie; same-origin/proxied). The DoD's end-to-end proof.

## Decision 5 — Store: realized slices vs the still-simulated remainder

The realized slices (`agents`, `providers`, `models`, `session`, onboarding) initialize **empty** and load from the API on `ready` (not from `SEED_*`) — otherwise seeded rows would flash as real data. The still-simulated slices (`requests`, `stats`, `chart`, `tiers`, `rules`, `limits`, `channels`) and the 4s live feed stay for the deferred pages, but are kept in a clearly-separated `demo` namespace / behind their pages, and the **Overview / Requests / Costs / Routing / Limits / Notifications pages are visibly marked "preview — simulated" (or their write-controls disabled)** so nothing simulated masquerades as operational. The `Sidebar`/`Overview` stop deriving live numbers from the now-real (initially empty) `providers` array where that would misrepresent state.

## Decision 6 — Test strategy

- **Injection seam:** the app currently imports the process-wide `app` singleton, so `createAppStore(fakeClient)` alone can't reach component tests. Introduce a **Solid context** (`AppProvider` holding the store) that `App`/pages consume, so `App.test.tsx` renders `<AppProvider store={createAppStore(fakeClient)}>` and supplies a `FakeApiClient`. The `live={false}` seam stays.
- **Frontend (Vitest + happy-dom):** a `FakeApiClient` records calls and returns canned responses; new specs cover the auth gate (me-200 → shell; me-401 → login; sign-in → ready; **sign-out under auto-login stays ready**), agents CRUD + key-shown-once + key-cleared-on-dismiss, provider create→test→sync + kind mapping + custom/local price edit (+ the custom/local-only rejection), and the onboarding walk incl. failure branches. Existing `appState.test.ts`/`App.test.tsx` are updated to inject the fake client and drop simulator assertions for realized slices.
- **Backend (Jest e2e):** `GET /api/me` returns the principal under cookie auth **and** under localhost auto-login (no cookie), 401 otherwise; `GET /api/login-config` (booted through the real `mountAuth` ordering) lists exactly the configured OAuth providers and is public; `PATCH /api/models/:id` sets prices, is **owner-scoped** (cross-tenant by-id → 404) and reflected in `SafeModel`; a cross-tenant/IDOR case for the new model route.

## Risks / trade-offs

- **Half-real dashboard.** Management is real; observe/config pages stay simulated (deferred #19/#20). Usage/spend columns on the real Agents/Providers pages read "—" until #17. Documented; the alternative (block #18 on #17) is worse for shipping the onboarding DoD.
- **Auto-login without a cookie.** On self-host loopback the SPA is authorized with no session cookie, so logout is a no-op there and `get-session` is null — `/api/me` is the single source of truth, and the account panel notes the auto-login mode. In cloud/non-loopback the normal cookie flow applies.
- **Model-own price vs effective-dated catalog.** Custom/local prices are set on the `model` row (mutable current price, top precedence) rather than an effective-dated catalog row (whose `model_key` is null for custom/local). Immutability of *past* cost is preserved by RequestLog snapshots; only future requests see the new price. The known-model catalog override stays effective-dated.
- **New frontend fetch layer.** No HTTP client/router exists; using bare `fetch` (no new deps) keeps the bundle lean but means hand-rolled error/loading handling — bounded to the `ApiClient` + per-slice store state.
- **One migration? No.** No schema change — `getIdentity` reads existing columns; model price columns already exist. No new deps, no migration.
