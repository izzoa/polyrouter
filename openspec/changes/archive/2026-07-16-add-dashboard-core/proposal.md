# Proposal: add-dashboard-core

## Why

The dashboard is the product's face and the first half of its one-liner ("connect an agent → connect providers → configure → observe", §2). The SolidJS SPA already exists — built out-of-band (`add-dashboard-prototype`) from the approved design — but it runs entirely against an **in-memory simulator** (`src/data/` + `src/state/appState.ts`): no `fetch`, no auth gate, no session. Every backend surface it needs is already built and session-guarded (#3 auth, #7 providers, #8 pricing, #9 routing) and the SPA is already served on one port with `/api`+`/v1` proxied in dev — so this change is mostly **frontend plumbing: swap the simulator for real HTTP behind the store↔data seam**, plus the three small backend endpoints the dashboard needs that don't exist yet.

This is #18 `add-dashboard-core` — the **management** slice: auth + Agents + Providers + the 3-step onboarding, meeting the DoD "a new user can sign up, mint an agent key, add a provider, and hit the proxy end to end." The **observe** pages (Overview/Requests/Costs, need #17 analytics) and the remaining **config** pages (Routing/Limits/Notifications) stay on the simulator behind the same seam until #19/#20.

## What Changes

**Backend — fill three dashboard gaps (small, owner-scoped, no new deps):**

- **`GET /api/me` (session-guarded).** Returns the current principal's identity for the account UI + admin gating: `{ userId, email, name, role, mode }`. Because the global `SessionGuard`'s localhost auto-login sets `req.principal` for `/api/*` **without issuing a cookie** (so Better Auth `get-session` is null on self-host loopback), this session-guarded endpoint is the SPA's single reliable "am I authorized, and as whom" call under **both** cookie auth and auto-login.
- **`GET /api/login-config` (`@Public()`).** The login page's bootstrap: `{ mode, emailPassword: true, oauthProviders: string[] }` — the OAuth providers list reflects which id/secret env pairs are actually configured (google/github/discord), so the login page renders only working buttons. **Named outside the `/api/auth*` prefix** that the raw Better Auth middleware intercepts before Nest (an `/api/auth-config` route would 404).
- **`PATCH /api/models/:id` (session-guarded, owner-scoped, custom/local only) + prices on `SafeModel`.** The §7.7 "user-entered prices for custom/local models" path: set the model-own `inputPricePer1m`/`outputPricePer1m`/`isFree` (the highest-precedence source in `resolveModelPrice`) through the existing owner-scoped `ModelAccessor.update` (invariant 5). The service **rejects (422) known-provider (`api_key`/`subscription`) models** — for those the bundled catalog is authoritative — and validates pricing **atomically** (a price *pair*, or *free* normalized to 0/0/free; a half-set price is 422). `SafeModel` (GET `/api/models`) gains the price fields. Editing a model's *current* price never rewrites past spend — the recorder snapshots prices at completion (invariant 4). (The catalog override for *known* models already exists at `POST /api/pricing/:modelKey/override`, admin+selfhosted; the dashboard surfaces catalog prices read-only.)

**Frontend — make the management core real (the bulk of the work):**

- **API client seam.** A typed `fetch` layer (`credentials: 'include'`, relative `/api/...` so it's same-origin in prod and Vite-proxied in dev) that the store's actions call, injectable so Vitest can supply a fake (no fetch mocking exists today). The store keeps the same field/action names — pages read `state.*` and call `app.*` unchanged where possible; the seam is the store↔data boundary.
- **Auth bootstrap + gate.** On load, call `GET /api/me`: **200 → render the dashboard** (self-host loopback lands straight in, as admin, via auto-login); **401 → render the login/signup page** (email/password via `/api/auth/sign-in|sign-up/email`, OAuth buttons from `/api/login-config` via `/api/auth/sign-in/social`). A **logout** action (`POST /api/auth/sign-out`) and a real account panel (identity + role + mode) replace the hardcoded `admin@localhost`.
- **Agents page → real CRUD.** List/create/rotate/delete against `/api/agents`; the **key + snippet are shown once** from the create/rotate response (server-built from the shared `connectionSnippet` at the instance's real base URL); add the missing delete action. Usage/spend columns (need #17) show "—" until analytics.
- **Providers page → real CRUD + custom/local pricing.** Add all four kinds (mapping the UI's `api|sub|custom|local` to the API's `api_key|subscription|custom|local`); reconcile the flow to the backend's **create-then-test/sync** order (UI currently tests before create); per-provider `test-connection`/`sync-models`; a model list with an **inline price editor for custom/local models** (`PATCH /api/models/:id`; local defaults to free); **subscription-kind ToS UX** — surface the reuse-a-flat-plan risk (§16) and nudge adding a pay-per-token fallback. Credentials stay write-only (never displayed; invariant 8).
- **Onboarding (Setup) → real end-to-end.** Step 1 `POST /api/agents` (mint + show key/snippet); Step 2 `POST /api/providers` → `sync-models` → assign the first synced model to the `default` tier (`PUT /api/routing/tiers/:defaultId/entries`, position 0); Step 3 a **"send a test message"** that POSTs `{model:'auto'}` to `/v1/chat/completions` with the minted key and shows the real proxied response — closing the DoD loop.

**Out of scope (deferred behind the same seam):** Overview/Requests/Costs analytics (#17/#19), Routing/Limits/Notifications config (#20), a live circuit-breaker/health read (#21), and bundling the Geist font locally (#22). Those pages keep the simulator; this change does not touch them beyond leaving them working.

Governing invariants: **5** (every dashboard read/write is owner-scoped — the new model PATCH goes through the central owner-scoped accessor, no unguarded by-id), **4** (model-own price edits don't recompute historical cost; snapshots are immutable), **8** (provider/agent credentials are write-only and shown-once; never redisplayed or logged), **6** (provider base_url SSRF validation is unchanged, server-side). Auth's two planes stay separate: the dashboard is cookie/session (`/api`), the proxy stays agent-key (`/v1`).
