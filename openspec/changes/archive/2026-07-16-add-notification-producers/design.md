# Design: add-notification-producers

## Context

Reuses #15a's `NotificationService.emit(event)` (non-blocking, TTL-deduped, sanitized) + its `NotificationEvent`/`EventScope`/`renderEvent` contract and SSRF-guarded `deliverSmtp`; #11's `request_log` + #14's `request_attempt` (per-cascade-attempt `cost`); #6's Redis circuit breaker (`breaker.ts`, atomic Lua, per-instance in-memory fallback); #3's Better Auth `createAuth` (the `sendResetPassword` no-op stub); #1's `registerConfig`; #10's atomic Redis counters. New producer code lives in `control-plane/src/producers/`; the only data-plane touch is an **additive** breaker return value + callback. Governing invariants: **11** (every producer is fire-and-forget — no producer/notification/mailer failure or latency reaches a request, budget check, or reset), **1** (breaker/proxy change never gates a request on a notification), **10** (spike uses an atomic Redis counter, correct across instances), **6** (reset SMTP host SSRF-validated + IP-pinned), **8** (creds/tokens never logged), **5** (events owner-scoped).

## Decision 1 — `provider_down`: surface the breaker open-transition (additive), emit from the proxy, best-effort

The breaker is the only correct emit site: the transition is applied atomically by one completion, and provider **name + owner** are only resolvable where the request's `principal` + `ProviderRow` live (a `cb:*` poller can't resolve them without an unscoped by-id lookup — §11.1 forbids it). This yields **one best-effort event *emission* per shared open transition** (an alert, not a ledger): the emit is after the atomic transition, so a crash in between drops it — acceptable, cheaper than an outbox. (Channel *delivery* is #15a's at-least-once BullMQ retry + 1 h fan-out dedup — not exactly-once; the guarantee here is on emission, deduped per provider/hour.)

- **`BreakerStore.complete` returns `BreakerCompletion = { justOpened, generation, openedAt }`** instead of `void`. `justOpened = priorState !== 'open' && nextState === 'open'` (covers closed→open and half_open→open). **State math is unchanged** — `applyComplete`/`COMPLETE_LUA` already compute it; `justOpened` is derived from pre/post state. The Lua must return the **tuple on every path**, including the current early `return 0` for a stale token / neutral outcome (return `{0, generation, openedAt}` there); the parser reads a 3-tuple.
- **Only the shared (primary) store alerts.** When Redis admission fails the breaker uses a per-instance in-memory fallback (`breaker.ts:323`), so N instances could each open locally. `CircuitBreaker.complete` returns `justOpened` **only when the completion ran on the primary store** (`token.isPrimary`), else `justOpened:false` — so a Redis outage doesn't fan out N duplicate alerts (and notifications are degraded then anyway, per invariant 11). A caught store error also returns `justOpened:false` (never alert on a breaker fault).
- **`withBreaker`/`withBreakerStream` gain `onOpen?: (providerId, info) => void`**, invoked once when a `complete` reports `justOpened`, wrapped so it can never throw into the call path; in the stream wrapper it hangs off `settle`.
- **Chain walkers thread a per-request `onOpen`** (on the existing `ctx`/`opts` arg) and forward it per attempt: `withBreaker(breaker, attempt.providerId, fn, ctx.onOpen)`.
- **`ProxyService` supplies the callback**, closing over `buildBundle`'s `providerId → { name, ownerUserId }` map: `(providerId) => { const p = rows.get(providerId); if (p) this.producers.providerDown(providerId, p.name, p.ownerUserId); }`, which does `void emit({ type:'provider_down', scope:{ ownerUserId, providerId }, fields:{ providerName } })`. **No `lifecycleId`**: `generation` is a concurrency version (it bumps on half-open admission *and* on a failed probe — `breaker.ts:69,132`), so using it would re-alert on every probe; instead dedup on `scopeKey=(type, owner, providerId)` within `provider_down`'s **1 h window** → at most one alert per provider per hour, and a re-open within the window is suppressed (matches the spec scenario). `void` + non-throwing `emit` ⇒ the hot path is untouched.

## Decision 2 — `request_failures_spike`: atomic Redis counter (no log-writer race)

The RequestLog write path is **async-batched** (`record()` enqueues a draft; `LogWriter` flushes to Postgres ~1 s later), so a DB `count` right after recording an error would miss the triggering row and its batch. Instead, count in Redis at the moment the error is known:

- On a recorded **error** (from **any** final-status-`error` path — buffered, pre-/post-commit stream, cascade), `ProxyService` fire-and-forgets `void this.producers.onRequestFailed(principal)`. The producer runs **one atomic Lua op** on `spike:<owner>:<bucket>` (`bucket = floor(now / windowMs)`): `INCR`, set `PEXPIRE = windowMs` only when the value is 1, return the new count `n` — so increment + expiry can't split (no immortal bucket, no lost-expiry suppressing the alert). When `n === NOTIFY_FAILURE_THRESHOLD` it emits `request_failures_spike` (`fields:{count:n}`, `lifecycleId=bucket`). `INCR` is atomic across instances (invariant 10) → the threshold-th error anywhere emits once per window. This is an **epoch-aligned tumbling window** (buckets don't slide) — simple and cheap; a burst straddling a boundary can split across two adjacent buckets (acceptable for an anti-spam alert). Threshold/window are validated positive. O(1), off the hot path, no DB query/index/writer-race.

## Decision 3 — `weekly_spend_summary`: narrow system reader on a single-run schedule

- **`WeeklySpendReader.weeklySpendByOwner(start: Date, endExclusive: Date): Promise<{ ownerUserId, total }[]>`** — a **narrow, scheduler-only capability** **constructed inside `DatabaseModule`** (which alone holds the private `DRIZZLE` handle) and exported as a single `WEEKLY_SPEND_READER` token; it is NOT added to `PersistencePort`/`PersistenceFacilities`, so the central "no unscoped read" safeguard stays intact for every other importer, and the raw drizzle handle is never re-exported. It sums **both cost ledgers** over the **bounded half-open interval** `[start, endExclusive)` on `request_log.createdAt` — `sum(coalesce(request_log.cost,0))` **plus** `sum(coalesce(request_attempt.cost,0))` for attempts joined to their parent `request_log` by `requestLogId` (so both ledgers share one period + reuse the `request_log.createdAt` / attempt indexes) — grouped by `ownerUserId`. Null costs coalesce to 0 (known spend). Cross-owner by construction, but each owner is emitted only their own row (invariant 5).
- **Scheduler = a BullMQ Job Scheduler** (`WeeklySummaryScheduler`, own `producers` queue + worker, dedicated ioredis connections, `Queue`/`Worker` error listeners + deadline-bounded shutdown — the #15a robustness patterns). On bootstrap, reconciliation is **fail-open**: `if (NOTIFY_WEEKLY_ENABLED) upsertJobScheduler('weekly-summary', { pattern: NOTIFY_WEEKLY_CRON, tz:'UTC' }, { name:'weekly_summary' }) else removeJobScheduler('weekly-summary')`, wrapped **deadline-bounded + caught + retried in the background** so a down Redis can **never delay or fail boot** (invariant 1 — Layer 0 must not be gated on notification infra); a boot-with-Redis-down test covers it. `upsertJobScheduler` (BullMQ 5) updates in place so a cron change / repeated boot doesn't duplicate the schedule; `tz:'UTC'` so every instance computes the same occurrence. Execution is **at-least-once** (a stalled job can re-run) with **idempotent effects**: the worker derives an occurrence key + `[start,end)` from the scheduled timestamp (`job.opts.prevMillis`, *not* `job.timestamp` = creation time), and `await`s each `emit` (`lifecycleId` = occurrence key, so a re-run **within `weekly_spend_summary`'s dedup retention** collapses to one summary per owner — realistic BullMQ stalls are seconds/minutes, well inside it). A Redis-down fire is a skipped occurrence (best-effort, no backfill). **Fail-open retry hygiene** (task 5.3): a single background backoff loop, no overlapping reconciliation, unref'd timer, cancelled on shutdown. Unknown-price rows (`cost` null) coalesce to 0 and the summary is rendered as **"Known spend"** (not "Total spend") so an all-unknown owner isn't shown a misleading `0` total — a one-line wording change to #15a's `renderEvent`.

## Decision 4 — password-reset email + server-wide SMTP defaults (§12), non-blocking

- **Config**: register optional `SMTP_HOST`, `SMTP_PORT`, `SMTP_USER`, `SMTP_PASS`, `SMTP_FROM`, `SMTP_SECURE` (`none|starttls|tls`). `resolveSystemSmtp` → `SystemSmtp | undefined` (undefined unless `SMTP_HOST` + `SMTP_FROM` set).
- **`SystemMailer.send(to, subject, text)`** builds an `SmtpConfig` and delegates to #15a's **`deliverSmtp`** — host SSRF-validated **at connect time, pinned to the resolved IP** (SNI kept), mode-gated policy, sanitized error codes. `deliverSmtp`'s `rt` param is narrowed to the `Pick<NotifyRuntime,'mode'|'allowedEndpoints'>` it actually reads.
- **Wire `sendResetPassword` non-blocking.** Better Auth **awaits** `sendResetPassword`, and `deliverSmtp` can wait on connect/greeting/socket timeouts, so the reset send MUST be **detached**: `sendResetPassword: ({ user, url }) => { if (!mailer?.configured) { logger.warn('reset email skipped: SMTP not configured'); return; } void mailer.send(user.email, 'Reset your polyrouter password', <body with url>).catch(() => logger.warn('reset email failed to send')); }` — returns immediately (the reset request never waits on or fails from SMTP), and **the token/url is never logged** (only fixed config-state strings). `AuthModule` imports the producers module for `SystemMailer`.

## Decision 5 — Module wiring & non-blocking discipline

`DatabaseModule` gains the `WEEKLY_SPEND_READER` provider (built on private `DRIZZLE`) and exports it. `producers.module.ts` imports `DatabaseModule` (for `WEEKLY_SPEND_READER`) + `RedisModule` (`REDIS_CLIENT` for the spike counter + scheduler) + `NotificationsModule` (`NotificationService`); provides `PRODUCERS_CONFIG`, `NotificationProducers`, `SystemMailer`, `WeeklySummaryScheduler`; exports `NotificationProducers` + `SystemMailer`. `ProxyModule` imports it → injects `NotificationProducers` into `ProxyService`; `AuthModule` imports it → `SystemMailer` into `createAuth`. Every producer call from the request/proxy/reset path is `void`-fire-and-forget over `emit`/a detached send, and scheduler reconciliation is fail-open, so no producer can add latency to or fail a request, fallback, reset, or **boot** (invariants 11/1).

## Risks / trade-offs

- **Breaker surface change** touches the data-plane hot path (#6) + `core.ts` + `proxy.service`. Mitigated: *additive* (return value + optional callback), state math + existing `complete` callers unaffected (the Lua's old `1`/`0` return was discarded), callback `void`/non-throwing, primary-store-only + best-effort. It changes an internal (not published) `BreakerStore` interface. Covered by breaker unit tests (justOpened once per transition, never on success/neutral/stale/fallback; Lua all-paths parity) + a proxy test (trip → one `provider_down`, down producer never fails the request).
- **provider_down is one best-effort *emission* per shared transition** (post-transition emit, primary-store only, per-provider-per-hour dedup); *delivery* is #15a's at-least-once retry. Stated, not hidden; an outbox is out of scope for an alert.
- **Spike is a Redis counter**, not a live RequestLog scan — avoids the async-writer race + DB pressure during an outage; the tradeoff is it counts *recorded* errors (an epoch-aligned tumbling window), documented.
- **Weekly aggregate is unscoped by necessity** but built in `DatabaseModule` behind a narrow scheduler-only `WEEKLY_SPEND_READER` token (raw drizzle never re-exported), per-owner-partitioned output + a cross-tenant test; bounded `[start,end)` interval; includes both cost ledgers; null costs → known spend.
- **Scheduler** uses a BullMQ Job Scheduler — **at-least-once** execution (stalled re-run tolerated) with **idempotent, occurrence-keyed** effects; reconciliation is **fail-open** so it never gates boot; a Redis-down fire is a skipped occurrence (best-effort).
- **No new heavy deps** — reuses `bullmq`/`nodemailer`.
