# Proposal: add-notification-producers

## Why

#15a shipped the notification **delivery core** ‚Äî the `NotificationService.emit(event)` contract, TTL dedup, and sanitized async fan-out ‚Äî but nothing yet *calls* `emit`, and Better Auth's `sendResetPassword` is still a no-op stub (deferred here from #3). This change wires the real **producers** for the three non-budget event types (`provider_down`, `request_failures_spike`, `weekly_spend_summary` ‚Äî `budget_*` are #16) and the **password-reset system email**, completing ¬ß10.1's "what triggers a notification." Every producer emits through #15a's non-blocking `emit`, so a slow/failing channel never touches the request path (invariant 11); the breaker change is additive (no state-math change); the reset mailer reuses #15a's SSRF-guarded, IP-pinned SMTP send (invariants 6/8).

## What Changes

- **`provider_down` ‚Äî surface the breaker open-transition (data-plane #6).** The Redis circuit breaker (`packages/data-plane/.../breaker.ts`) only *stores* state today; `complete()` returns `void`. Make the store/`CircuitBreaker.complete` return `{ justOpened, generation, openedAt }` (additive; state math unchanged; the Lua's discarded `1`/`0` return becomes a tuple on every path) and give `withBreaker`/`withBreakerStream` an optional `onOpen` callback the fallback-chain walkers (`core.ts`) thread per attempt. `ProxyService`, holding the `principal` + the attempt's `ProviderRow` (name, owner), supplies an `onOpen` that **fire-and-forget emits** `provider_down` scoped to the provider's owner. `justOpened` is true **only on the shared/primary store** (a Redis-outage fallback open doesn't fan out N alerts), so this is **one best-effort *emission* per open transition** (channel *delivery* is #15a's at-least-once retry, not exactly-once); dedup is on `(owner, providerId)` within the 1 h window ‚Äî **not** `generation` (a concurrency version that changes per probe, so it would re-alert) ‚Äî giving at most one alert per provider per hour.
- **`request_failures_spike` ‚Äî atomic Redis counter (invariant 10).** The RequestLog write path is async-batched, so a DB count right after an error would miss the triggering row. Instead, on a recorded chain **error** `ProxyService` fire-and-forgets a spike check that `INCR`s a windowed per-owner Redis counter (`spike:<owner>:<bucket>`, `PEXPIRE` the window) and emits `request_failures_spike` when it **equals** the threshold ‚Äî atomic across instances, O(1), off the hot path, no DB query/index/race. Dedup 15 min, `lifecycleId` = window bucket.
- **`weekly_spend_summary` ‚Äî scheduled per-owner rollup.** Add a **narrow, scheduler-only `WeeklySpendReader.weeklySpendByOwner(start, endExclusive)`** built inside `DatabaseModule` and exported as a `WEEKLY_SPEND_READER` token (NOT the general persistence seam; raw drizzle never re-exported) that sums **both** cost ledgers over the bounded interval ‚Äî `request_log.cost` **plus** `request_attempt.cost` (cascade, #14), null ‚Üí 0 ‚Äî grouped by owner; and a **BullMQ Job Scheduler** (`upsertJobScheduler`, fixed id + UTC, removed when disabled, **fail-open** reconciliation so it never gates boot) whose **at-least-once, occurrence-keyed idempotent** handler derives `[start,end)` from `job.opts.prevMillis`, awaits its emits, and emits one `weekly_spend_summary` per owner. Cross-owner by construction but per-owner-partitioned output, so no tenant sees another's total.
- **Password-reset email + server-wide SMTP defaults (¬ß12), non-blocking.** Register the optional server-wide `SMTP_HOST`/`SMTP_PORT`/`SMTP_USER`/`SMTP_PASS`/`SMTP_FROM`/`SMTP_SECURE`. Add a `SystemMailer` that sends via #15a's **SSRF-guarded, IP-pinned `deliverSmtp`**. Rewire Better Auth's `sendResetPassword({ user, url })` to **detach** the send (`void mailer.send(...).catch(log)`) so the reset request never waits on or fails from SMTP timeouts (Better Auth awaits the hook ‚Äî invariant 11) ‚Äî **never logging the token/url**; if SMTP is unconfigured, log-and-skip (the token flow still works). Reset-endpoint rate-limiting is #3's (¬ß11.3), unchanged.

Governing invariants: 11 (producers are async/non-blocking ‚Äî a producer failure never stalls a request, a budget check, or the reset flow), 10 (the spike counter is atomic in Redis, correct across instances), 6 (the reset SMTP host is SSRF-validated + pinned like every server-fetched target), 8 (channel/SMTP creds encrypted, tokens/secrets never logged), 5 (`provider_down`/spike are owner-scoped from the request principal; the weekly rollup emits per-owner). Layer-0 reliability (invariant 1) is untouched ‚Äî the breaker change adds a return value + a fire-and-forget callback, never gating a request on a notification.
