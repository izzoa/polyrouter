# Design: add-packaging

## Context

Current state: `npm run build` (turbo) produces `packages/control-plane/dist` (with migrations copied in), `packages/{shared,data-plane}/dist`, and `packages/frontend/dist`; `npm start` runs `node packages/control-plane/dist/main.js` with `NODE_ENV=production`. In production `configureSpa` serves the SPA from `join(__dirname, '../../frontend/dist')` and falls back every non-`/api`/`/v1` GET to `index.html` — so the health probe must target the REAL route, **`/api/health`** (bare `/health` returns the SPA shell with a misleading 200). Boot validates the full registered env fail-fast; migrations run in `DatabaseModule.onModuleInit` before listen (Drizzle's migrator takes no advisory lock — single app replica only); `enableShutdownHooks` drives the #12 stream drain, #11 writer flush, and #21 span flush on SIGTERM. Secrets policy (§12): `BETTER_AUTH_SECRET`, `API_KEY_HMAC_SECRET`, `PROVIDER_CREDENTIAL_KEY`, `NOTIFY_CREDENTIALS_SECRET` are 32-byte hex, REQUIRED whenever production or non-loopback-bound. `DATABASE_URL`/`REDIS_URL` have dev-compose defaults. The #15a SSRF posture rejects a private-range `APPRISE_API_URL` unless `NOTIFY_ALLOWED_ENDPOINTS` (port-bounded allowlist) admits it. `docker-compose.dev.yml` (postgres+redis, no explicit `name:` — its implicit project name is the repo DIRECTORY name, i.e. `polyrouter`) stays as dev infrastructure. The repo has an `.npmrc` (`engine-strict=true`) that `npm ci` needs in the build context. The frontend currently loads Geist from Google Fonts at runtime; the Phase F note in TODOS.md assigns bundling it locally to #22. The dependency tree has no REQUIRED native modules (BullMQ's optional `msgpackr-extract` has a pure-JS fallback), so alpine/musl is viable.

## Goals / Non-Goals

**Goals:**
- `docker compose up` from a checkout (or `install.sh` on a clean machine) yields a working instance on one published port: SPA + `/api` + `/v1` + `/api/health` + `/metrics`, migrations applied, first sign-up = admin — **published on loopback by default** (claiming the admin account happens before any network exposure).
- The container honors the platform lifecycle: healthcheck on `/api/health`, SIGTERM → graceful drain within the grace period (proven with an in-flight stream), non-root runtime.
- Secrets generated once, never committed, never echoed, never silently rotated; `.env` is the single operator surface.
- The packaged SPA is self-contained: no third-party runtime fetches (fonts bundled).
- Optional Apprise service that actually works when enabled (allowlist wiring documented), without burdening the default `up`.

**Non-Goals:**
- Registry publishing / CI release automation (the `image:` tag is forward-compatible); TLS/reverse-proxy termination (documented pointer); multi-instance orchestration (explicitly documented as unsupported — the boot migrator isn't advisory-locked); backups automation.
- Any config-schema or backend behavior change.

## Decisions

1. **Image: `node:24-alpine`, multi-stage, monorepo layout preserved.** Stage `build`: copy manifests (`package.json`, `package-lock.json`, `turbo.json`, **`.npmrc`**, `packages/*/package.json`) → `npm ci` (cached layer) → copy source → `npm run build` → `npm prune --omit=dev` (npm 11 is workspace-aware; the smoke boot is the correctness proof, and the runtime stage copies only `node_modules` + `dist` + manifests so any pruning residue costs size, not behavior). Stage `runtime`: `ENV NODE_ENV=production`, copy root `package.json`, pruned `node_modules`, each `packages/*/package.json` + `dist` preserving `/app/packages/...` (workspace resolution, `configureSpa`, bundled migrations all unchanged). `USER node`, `EXPOSE 3001`, exec-form `CMD` (Node is PID 1; SIGTERM → Nest shutdown hooks — the drain path). `HEALTHCHECK CMD wget -qO- http://127.0.0.1:3001/api/health || exit 1` (busybox wget ships with alpine).
2. **Compose topology (`name: polyrouter-selfhost`).** An explicit, distinct project name — the dev compose's implicit project is the directory name (`polyrouter`), so an unnamed product compose in the same directory would collide with it; `install.sh` additionally passes `-p polyrouter-selfhost` so an ambient `COMPOSE_PROJECT_NAME` can't defeat the isolation. `app`: `build: .` + `image: polyrouter:latest`, **`ports: ["${POLYROUTER_HOST:-127.0.0.1}:${POLYROUTER_PORT:-3001}:3001"]`** — loopback publish by default: an all-interfaces default would expose the unauthenticated sign-up (first account = admin) and `/metrics` to the network on first boot; going public is an explicit `.env` change made AFTER claiming admin, documented with the reverse-proxy guidance. `depends_on` postgres/redis `condition: service_healthy`, `stop_grace_period: 45s` (comfortably over the stream-drain budget), `restart: unless-stopped`. Compose-owned env: `BIND_ADDRESS=0.0.0.0` (correct INSIDE the namespaced container; the §12 loopback default would make the published port dead), `NODE_ENV=production`, `MODE=selfhosted`, `DATABASE_URL=postgresql://polyrouter:${POSTGRES_PASSWORD}@postgres:5432/polyrouter`, `REDIS_URL=redis://redis:6379`, `BETTER_AUTH_URL=${APP_URL:-http://localhost:3001}`. Secrets + `POSTGRES_PASSWORD` interpolate from `.env`. Optional vars (`METRICS_ENABLED`, `OTEL_ENABLED`, `OTEL_EXPORTER_OTLP_ENDPOINT`, `APPRISE_API_URL`, `NOTIFY_ALLOWED_ENDPOINTS`, OAuth client ids/secrets) pass through as **bare keys** in the environment list — an unset var stays UNSET in the container (several of these reject empty strings, so `${VAR:-}` interpolation would break boot).
3. **Apprise wiring that actually passes the SSRF guard.** `apprise` (`caronc/apprise:latest`, config volume) sits under `profiles: ["apprise"]` and is not depended on by `app`. Enabling it is a documented two-line `.env` step: `APPRISE_API_URL=http://apprise:8000` **and** `NOTIFY_ALLOWED_ENDPOINTS=<allowlist entry for the apprise host:8000>` (exact syntax per the #15a allowlist parser, pinned during implementation) — without the allowlist the boot guard rejects the private-range service address by design (§10.1). This keeps the egress decision explicit and the guard intact rather than compose silently pre-authorizing a private endpoint.
4. **`install.sh` (POSIX sh, `set -eu`).** Preconditions with clear failures: `curl`, `tar`, `docker info` (daemon reachable), and **Compose v2** (`docker compose version` — no legacy `docker-compose` fallback). Fetches ONE pinned source archive (codeload tarball of the default branch today; a release tag once releases exist) and uses the compose file FROM that archive — compose and build context are always the same commit; when run inside a checkout it uses the working tree instead. First run writes `.env` under `umask 077` to a temp file and atomically renames it: four `openssl rand -hex 32` secrets + a random `POSTGRES_PASSWORD` (a `/dev/urandom` fallback generator when openssl is absent) + a sibling `.gitignore`. An existing `.env` is NEVER overwritten or amended-with-rotation (rotating the two encryption keys would orphan encrypted provider/channel credentials; the script says so). Re-extraction goes through a staging directory (no stale files from deleted paths). Then `docker compose -p polyrouter-selfhost up -d --build` and print the URL + first-sign-up note — no secret values ever printed. The README notes the pipe-to-sh trust posture and the inspect-first/clone alternative.
5. **Local fonts (TODOS Phase F note → #22).** Vendor the Geist + Geist Mono woff2 weights the CSS actually uses into `packages/frontend/public/fonts/` with their OFL license file; replace the Google Fonts `<link>`s in `index.html` with a local `@font-face` stylesheet. Asset-only change (no logic); the packaged SPA then loads with zero third-party runtime fetches — self-host posture and offline correctness.
6. **Verification is a scripted smoke test** (`scripts/selfhost-smoke.sh`), not Jest — repeatable DoD evidence, run against throwaway volumes and a non-conflicting host port (project name isolation keeps the dev compose untouched):
   - healthchecks green; `GET /api/health` 200 **JSON** (content-type asserted — the SPA fallback would be HTML); `GET /` serves the SPA shell referencing only local assets; `GET /metrics` 200 Prometheus text.
   - sign-up #1 via the auth API → admin (asserted via the account API/DB role); sign-up #2 → not admin.
   - **drain proof with a live stream:** a loopback SSE stub exec'd inside the app container (`kind: local` provider — the self-host loopback SSRF exception), an agent key minted via the API, a slow streamed completion held open, then `docker stop` — assert the client's stream terminates cleanly, the container's **inspected** `.State.ExitCode` is 0 and `.State.OOMKilled` false (the compose command's exit code proves nothing), and the shutdown logs show the drain path.
   - **metadata-only proof:** the streamed prompt carries a sentinel; `psql` over the request rows asserts the sentinel appears nowhere (the #11 e2e already proves this in-process; this re-proves it for the packaged artifact).
   - install-script pass in a temp dir: first run creates `.env` (600, four distinct hex secrets) and boots; second run leaves `.env` byte-identical and re-applies.

## Risks / Trade-offs

- **Build-from-source compose** makes first `up` slow. Accepted for a registry-less baseline; the manifest→`npm ci`→source layer split keeps rebuilds incremental; `image:` is the forward path.
- **Alpine/musl**: no required native deps today (BullMQ's `msgpackr-extract` is optional with a JS fallback — the smoke pass exercises the queue path by booting notifications); a future required native dep fails the image build loudly, at which point the base flips to `-slim`.
- **`POSTGRES_PASSWORD` is initialization-only**: changing it in `.env` after the volume exists does NOT rotate the database role password (documented; operators rotate in postgres first).
- **Single replica only**: the boot migrator has no advisory lock; `--scale app` is documented as unsupported at this tier.
- **`BETTER_AUTH_URL` correctness** depends on the operator's real origin when exposed (`APP_URL` in `.env`, mandatory in the reverse-proxy example).
- **Smoke test host assumptions** (curl, docker, psql via the postgres container): it's an operator/dev tool run on demand, not CI, and says what it needs.
