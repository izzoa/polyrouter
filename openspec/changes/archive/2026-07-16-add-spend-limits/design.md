# Design: add-spend-limits

## Context

Reuses: #11's `request_log` + #14's `request_attempt` (immutable per-call `cost`, USD `number|null`, computed authoritatively in the batched `LogWriter` via `PricingService.resolveForModel`/`resolveUsage`/`computeCost`); #6's bounded/fail-fast Redis (dedicated connection, `structural-baseline.store` idea); #15a/#15b's `NotificationService.emit` (non-blocking) + the stubbed `budget_alert`/`budget_block` templates/dedup + `NotificationProducers` (already injected into `ProxyService`) + the BullMQ Job-Scheduler pattern; #2's `OwnedRepository` + `SessionGuard`. The proxy's `completion`/`stream` carry `principal` + `agentId`. Governing invariants: **10** (Redis-atomic counters, no per-instance drift), **11** (alerts fire-and-forget — never block a request/enforcement), **1** (Layer 0 not gated on budget/notification infra), **5** (tenant-scoped), **4** (immutable snapshot cost), **8** (no bodies/secrets).

**Core model — the counter is a reconciled read-model, not a live increment.** The authoritative per-request cost only exists later, in the batched `LogWriter` (effective-dated + cache-token pricing + resolved usage); a proxy-side estimate drifts. Rather than couple a live increment into the writer (which then double-counts against reconciliation and needs per-row idempotency markers), the **scheduler is the sole writer of the spend counter**: every `BUDGET_SCHED_CRON` interval it recomputes each active budget's current-period spend from RequestLog (both cost ledgers) and **`SET`s** the Redis counter. The proxy's block check reads that counter. This makes the counter **exactly** `request_log.cost + Σ request_attempt.cost` for the period (no estimation drift, no double-count, no race — a single writer), at the cost of **enforcement latency = the reconcile interval** (tunable; default short). This is an **asynchronously-metered postpaid soft cap** over **UTC calendar windows that reset at the boundary** (this delta reinterprets §10's "rolling window" as the resetting calendar period its own "until reset" wording implies; the root `spec.md` §10 wording is updated to match — a task). A pre-authorized hard cap and a sliding window are deferred.

## Decision 1 — Data model + CRUD

`budget` table (name avoids the `limit` keyword), owner-scoped like #15a's `notification_channel`: `id`, `ownerUserId`/`orgId`, `name`, `scope` (`global`|`agent`), `agentId` (nullable — set iff `scope='agent'`), `window` (`day`|`week`|`month`), `action` (`alert`|`block`), `amount` (`doublePrecision`, USD), `notifyChannelIds` (text csv), `enabled` (bool default true), `createdAt`; `index('budget_owner_idx')` + DB `check`s (`amount > 0` **and** `amount <= 1e9` so `round(amount×1e6)` stays a JS-safe integer; `scope`/`window`/`action` enums; `agentId` NOT NULL iff `scope='agent'`). Row type `BudgetRow`; `PersistencePort.budgets: OwnedRepository<…>` (owner forced from principal, no unscoped by-id — invariant 5). One migration (`0006`). CRUD at `/api/budgets`, class-validated (**including the merged state of a partial update** — a `scope`/`agentId` change is re-validated), `no-store` on mutations, **invalidating the local budget cache** for the owner on write. **Org scope deferred**; CRUD accepts only `global`/`agent`. A mid-period budget meters the whole current period (reconciliation sums from the period start) — documented.

## Decision 2 — Atomic Redis spend counters (integer micro-dollars, reconciled)

Money is integer **micro-dollars** (`µ$ = round(cost × 1e6)`) so values are exact; thresholds compare `µ$ ≥ round(amount × 1e6)`. One counter per **distinct (owner, scope, scopeId, window, period)**: key `budget:<owner>:<scope>:<scopeId>:<window>:<periodId>`, `scopeId` = `agentId` (agent) or `global`; `periodId` = the **UTC calendar period** (day `YYYY-MM-DD` / ISO week `YYYY-Www` / month `YYYY-MM`) computed in Node; `ttlMs` = ms to the period's UTC end **+ a grace**. The counter is **written only by the scheduler** (`SET` to the reconciled µ$ sum, + TTL) and **read** by the block check (`MGET`, missing = 0). A `SET NX PX` marker keys alert/block dedup. All budget Redis ops run on a **dedicated `REDIS_CLIENT.duplicate({ enableOfflineQueue:false, commandTimeout: BUDGET_REDIS_TIMEOUT_MS, maxRetriesPerRequest:1 })`** connection (so a timed-out request-path read fails fast and doesn't pile up during an outage). Because a single actor (the single-run scheduler) writes each counter, there is **no increment race and no double-count** — the counter equals the RequestLog period sum by construction.

## Decision 3 — Block enforcement in the proxy (bounded, named fail mode)

`BudgetService` (injected into `ProxyService`). At the **first statement** of `completion` and `stream` (both carry `principal`+`agentId`; streaming throws pre-commit so it renders cleanly):

```
const hit = await this.budgets.checkBlocked(principal, agentId);
if (hit) { this.budgets.notifyBlocked(principal, hit); throw budgetBlocked(hit); }
```

- `checkBlocked` reads the owner's budgets from a **capped, single-flight, short-TTL (10 s) in-memory cache** (a DB read only on a cold miss; invalidated locally on CRUD), filters `enabled && action==='block' && applies(b, agentId)` (`applies` = `scope==='global' || (scope==='agent' && agentId===b.agentId)`), `MGET`s the **distinct** counter keys once, then evaluates **every** matched budget against its own key's value + threshold (budgets sharing a key but with different amounts are all checked), and returns a **`BudgetHit` { budget, spentMicros, periodId, resetAt }** for the first over-threshold budget (or null). The hit's `spentMicros`/`resetAt` feed both the 402 message ("naming … its reset") and the `budget_block` fields.
- **Fail mode is a named, configurable contract** (`BUDGET_FAIL_OPEN`, default `true`). On a Redis fault/timeout (or a cold-cache DB failure): fail-open → allow (availability), or `BUDGET_FAIL_OPEN=false` → reject with `503 budget_enforcement_unavailable`. Never reject-all-silently or hang. (A degraded-enforcement metric is worth adding.) Invariant 1 governs the smart router degrading, not spend enforcement — either mode complies.
- `budgetBlocked(hit)` → `ProxyError(402, 'budget exceeded: <name> (resets <resetAt>)', 'invalid_request_error', 'budget_exceeded')`. Pre-`prepare()`, so a blocked request does zero routing/upstream work. **No spend increment in the proxy** — the scheduler is the sole counter writer (Decision 2/4).

Overshoot: enforcement is eventually-consistent within the reconcile interval — a request admitted before the next reconcile (and any concurrent/expensive one) can overshoot; the cross-instance shared counter then stops *subsequent* traffic once the reconcile reflects the crossing. Tighten the interval for tighter enforcement; a synchronous reserve-then-settle hard cap is deferred.

## Decision 4 — Scheduler (own queue): reconcile + alert (the sole counter writer)

A `BudgetScheduler` mirroring `WeeklySummaryScheduler` but on its **own dedicated BullMQ queue** (`budget-eval`, NOT the shared `producers` queue — two workers on one queue steal/complete each other's jobs), gated on `BUDGET_SCHED_ENABLED` (default **on** — this is the enforcement engine, not opt-in), every `BUDGET_SCHED_CRON` (default `* * * * *` = each minute; tunable tighter/looser for the enforcement-latency/DB-load trade-off), `tz:'UTC'`, fail-open bootstrap + conditional worker + bounded shutdown. It uses a narrow scheduler-only `BUDGET_READER` (built in `DatabaseModule` on private drizzle, token-only export): `listActiveBudgets()` and `spendMicrosFor(ownerUserId, agentId|null, start, endExclusive): number` — sums **both** ledgers as **row-level integer micros** (`sum(round(coalesce(cost,0)*1e6))`, the exact figure the block check compares — no sum-then-round), filtering each ledger by its own `created_at ∈ [start,end)`, and **joining `request_attempt` to its parent `request_log` only to attribute `agentId`** (attempts have no `agentId` column). Each occurrence evaluates the period at the job's scheduled `prevMillis − 1 ms` (so a run at exactly a boundary reconciles the **just-closed** period), and **groups active budgets by their distinct counter key** (one ledger scan + one `SET` per key, then evaluate every budget on that key):

- **Reconcile**: `SET` the counter to `µsum` (+ period-end-plus-grace TTL). Single writer ⇒ authoritative, race-free.
- **Alert**: for each `action='alert'` budget on the key, if `µsum ≥ toMicros(amount)` and `SET budget-alerted:<id>:<periodId> 1 NX PX <ttl>` wins (first crossing this period), emit `budget_alert` — **at most one per budget per period** (`lifecycleId=periodId` keeps a new period un-suppressed). Delivery is **best-effort** (one emit attempt per period; a Redis-down enqueue at that instant is a miss — consistent with overall outage degradation; a durable claim/outbox is deferred).

`budget_block` is emitted **inline** by `notifyBlocked` on the first block of a period, deduped by the same `SET NX` marker.

## Decision 5 — Producers, channel targeting, module wiring

- `NotificationProducers` gains `budgetAlert({ ownerUserId, agentId?, budgetId, periodId, name, spent, threshold, channelIds })` and `budgetBlock({ …, channelIds })` — each `void emit({ type, scope:{ ownerUserId, agentId?, limitId: budgetId, lifecycleId: periodId }, fields:{ limitName: name, spent: fmt, threshold: fmt }, channelIds })`, own try/catch. `NotificationEvent` gains an optional `channelIds?: string[]`; #15a's fan-out (`notify.queue`) **intersects** the owner's enabled+subscribed channels with `channelIds` when present (absent ⇒ all subscribed as today) — honoring §5's per-budget `notify_channel_ids`.
- `BudgetsModule` (imports `DatabaseModule` for the `budgets` repo + `BUDGET_READER`, `RedisModule`, `ProducersModule` for `NotificationProducers`): provides `BUDGETS_CONFIG`, the dedicated-connection `SpendCounter`, `BudgetCache`, `BudgetService`, `BudgetScheduler`, CRUD service; controller `BudgetsController`; exports `BudgetService`. `ProxyModule` imports it → injects `BudgetService`. Registered in `app.module`. **No writer/`SPEND_SINK` coupling** (the scheduler is the only counter writer), so `RecordingModule` is untouched.
- Every request-path budget touch is a **bounded, configurable-fail read** (block check) or a **`void` fire-and-forget emit** — no budget code adds latency to or fails a request, fallback, or block-enforcement (invariants 1/11); the cache keeps the hot path DB-free.

## Risks / trade-offs

- **Asynchronously-metered soft cap** — spend is reconciled from RequestLog every interval (default ~1 min, tunable), so `block` stops *subsequent* requests eventually-consistently within that interval; a small/tight budget under a burst can overshoot until the next reconcile. Tighten the interval (vs DB load) for tighter enforcement; a synchronous/reserve-then-settle hard cap is deferred. This trades the live-INCR's ~1 s latency for a single-writer counter that can't double-count or race — the correct trade for a budget.
- **Reconcile cost at scale** — recomputing all active budgets' spend each interval is N grouped `SUM`s (over `created_at`+`owner`-indexed rows for the current period); grouped by distinct key. Fine for modest N; batch/shard at scale (noted).
- **Fail mode is a deliberate, configurable contract** (`BUDGET_FAIL_OPEN`, default open). Operator's choice; local-shadow grace + a degraded-enforcement metric deferred.
- **Best-effort ledger + alerts** — RequestLog can drop rows under overflow (unmetered, and unrecoverable since never logged); a Redis-down moment can miss an alert for that period (at-most-one-attempt). Documented.
- **Calendar (resetting) windows** — reinterprets §10's "rolling" (root spec updated to match); a sliding window deferred; mid-period budgets meter the whole period.
- **Unpriced spend unmetered** — null-cost (local/unpriced) requests don't count toward budgets. Documented.
- **No new deps; one migration** (`budget`).
