# Proposal: add-spend-limits

## Why

Spend limits are half the product's one-liner ("route each call… with automatic fallbacks and **spend limits**"), and the last Phase-E capability: the `Limit` table (spec §5) plus §10's budgets — scoped global/agent over a day/week/month window, action **alert** or **block**. `block` must reject over-budget requests **in the proxy path** and be **correct across instances** (a shared Redis counter, not per-instance drift — the classic multi-instance budget bug, §3.2). `alert` fires **on a schedule** and emits `budget_alert`/`budget_block` into the notification pipeline #15a/#15b already built (those event types are stubbed and waiting). The budget check must add **negligible hot-path latency** and never let a failing notification channel block enforcement (invariant 11), while staying tenant-scoped (invariant 5) and atomic in Redis (invariant 10).

## What Changes

This is an **asynchronously-metered postpaid soft cap**: the spend counter is a **reconciled read-model** — the scheduler recomputes each budget's current-period spend from RequestLog every (short, tunable) interval and `SET`s the Redis counter (a single writer, so no double-count/race), and `block` rejects *new* requests once that counter crosses the threshold (eventually-consistent within the interval). Windows are **UTC calendar periods that reset at the boundary** (this delta reinterprets §10's "rolling window" as the resetting calendar period its own "until reset" wording implies; the root spec is updated to match).

- **`budget` table + CRUD (§5).** A new owner-scoped `budgets` table (name `budget`, avoiding the `limit` SQL keyword): `id`, `ownerUserId`/`orgId`, `name`, `scope` (`global`|`agent`; org deferred — `assertUserPrincipal` throws today), `agentId` (nullable; set for `agent` scope), `window` (`day`|`week`|`month`), `action` (`alert`|`block`), `amount` (USD threshold), `notifyChannelIds` (csv), `enabled`, `createdAt`; owner index, **DB `check` constraints** (`amount>0`; enums; `agentId` iff `scope='agent'`), one migration (`0006`). Class-validated, ownership-scoped CRUD at `/api/budgets` (the `OwnedRepository` + `SessionGuard` pattern), tenant-isolated, cache-invalidating on write. Added to `PersistencePort`.
- **Atomic Redis spend counters in integer micro-dollars (§10, invariant 10), reconciled from RequestLog.** A `SpendCounter` keyed `budget:<owner>:<scope>:<scopeId>:<window>:<periodId>` (`periodId` = the UTC calendar period) holding integer **µ$** (`round(cost×1e6)`, exact — no float drift). The **scheduler is the sole writer**: it `SET`s each counter to the authoritative both-ledger RequestLog sum (`request_log.cost + Σ request_attempt.cost`, computed once, in the writer) for the period every interval — so no proxy-side estimate (which would drift off the nullable model-level `AttemptMeta` prices) and no live-increment double-count/race. All budget Redis ops use a **dedicated fail-fast connection** (`enableOfflineQueue:false` + `commandTimeout`) so a request-path read can't pile up during an outage. Unpriced (null-cost) requests aren't metered.
- **Block enforcement in the proxy path.** At the **start of `completion`/`stream`** (both carry `principal`+`agentId`), `BudgetService.checkBlocked(principal, agentId)` `MGET`s the applicable **block**-budgets' counters (from a capped 10 s in-memory cache so the hot path is DB-free) and, if any is at/over its threshold, throws `ProxyError(402, budget_exceeded)` before any upstream work (streaming rejects pre-commit). The check is bounded; the **fail mode is a named, configurable contract** (`BUDGET_FAIL_OPEN`, default open → allow on a Redis fault for availability; `false` → `503 budget_enforcement_unavailable`). Invariant 1 governs the smart router degrading, not spend enforcement — so either mode complies.
- **Scheduler (own queue) = the sole counter writer + alerts + producers.** A `BudgetScheduler` on its **own dedicated BullMQ queue** (`budget-eval`, not the shared `producers` queue — two workers on one queue steal each other's jobs) using the #15b pattern (fail-open bootstrap, conditional worker, bounded shutdown, single-run) and a narrow cross-owner `BUDGET_READER` (built in `DatabaseModule` on the private drizzle handle) that sums both ledgers as **row-level integer micros** (joining attempts to their parent log for `agentId`). Grouping active budgets by distinct counter key, each occurrence **`SET`s** the counter to the reconciled sum (single writer ⇒ authoritative, race-free) and, for `action='alert'` at/over threshold, emits `budget_alert` deduped to **at most one per budget per period** via a Redis `SET NX` marker + `lifecycleId=periodId`. `budget_block` is emitted **inline** on the first block of a period (deduped). New `NotificationProducers.budgetAlert(...)`/`budgetBlock(...)`; the event carries the budget's `notifyChannelIds` and #15a's fan-out intersects them (honoring §5's per-budget channel selection).

Governing invariants: 10 (counters atomic in Redis, correct across instances — no per-instance drift), 11 (increment/alerts fire-and-forget; a failing channel never blocks a request or enforcement), 1 (Layer 0 not gated on budget/notification infra; the budget check is bounded with a named fail mode), 5 (tenant-scoped; block-check reads only the principal's budgets), 4 (spend uses the immutable snapshot cost, computed once), 8 (no secrets/bodies in budget events). Org-scoped budgets, a reserve-then-settle hard cap, a sliding window, and a budgets SPA are out of scope (deferred / #18/#20).
